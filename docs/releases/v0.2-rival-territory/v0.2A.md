# 2. Static content vs mutable GameState

Use **static definitions + mutable state overlays**.

This is the right call.

## Static content

These should live in content registries:

```ts
DistrictDefinition;
VenueDefinition;
RivalDefinition;
OperativeDefinition;
ActionDefinition;
EventDefinition;
```

Static examples:

```text
district name
district archetype
base district heat
base wealth
base secrecy
venue modifiers
venue tags
rival archetype
rival traits
```

## Mutable game state

These should live in `GameState`:

```text
district control
district current heat, if used
rival pressure
rival disposition
rival active/inactive status
venue controlling rival, if ownership can change later
```

For v0.2, I’d use this shape:

```ts
type GameState = {
  // existing fields...
  districts: Record<DistrictId, DistrictState>;
  rivals: Record<RivalId, RivalState>;
  recentActivity: RecentActivityEntry[];
};
```

Where:

```ts
type DistrictState = {
  id: DistrictId;
  control: number;
  heat: number;
};

type RivalState = {
  id: RivalId;
  pressure: number;
  disposition: number;
};
```

Then static definitions remain:

```ts
type DistrictDefinition = {
  id: DistrictId;
  name: string;
  archetype: DistrictArchetype;
  baseControl: number;
  baseHeat: number;
  wealth: number;
  secrecy: number;
  volatility: number;
  rivalId?: RivalId;
  venueIds: VenueId[];
  tags: string[];
};

type VenueDefinition = {
  id: VenueId;
  name: string;
  districtId: DistrictId;
  wealthMod: number;
  intelMod: number;
  heatMod: number;
  loyaltyMod: number;
  ruinMod: number;
  controllingRivalId?: RivalId;
  tags: string[];
};
```

This avoids mutating content definitions and keeps future save/load sane.

# 3. Does local district heat affect global Heat?

For v0.2:

> **Local district heat affects targeting modifiers, risk, and event weighting. It does not automatically roll up into global Heat.**

Global Heat remains the main loss pressure.

District heat is local texture and modifier state.

Use district heat for:

```text
risk calculation
target preview
event weighting
district display
future local consequences
```

Do not make it a second loss meter yet.

This keeps v0.2 from becoming overcomplicated.

---

# 4. Does local district heat move?

I would let it move, but lightly.

Not totally static.

If district heat never changes, it risks feeling fake. But if it becomes a full system, it adds tuning burden.

So v0.2 rule:

> **Actions that create positive global Heat also increase target district heat by a small amount.**

Simple rule:

```ts
if (order.target?.type === 'district' || order.target?.type === 'venue') {
  const districtId = resolveDistrictId(order.target);
  const globalHeatDelta = Math.max(0, resolvedEffects.heat ?? 0);

  districtState.heat += Math.ceil(globalHeatDelta / 3);
}
```

Examples:

```text
+12 global Heat → +4 local district Heat
+4 global Heat → +2 local district Heat
0 or negative Heat → no local Heat increase
```

Optional cooling:

```ts
districtState.heat = Math.max(definition.baseHeat, districtState.heat - 1);
```

Apply local cooling weekly after global drift.

Important: local heat should not drop below base heat in v0.2.

So:

```text
baseHeat = ambient danger of that district
currentHeat = baseHeat + recent trouble
```

This makes districts feel reactive without creating a new major strategic layer.

# 5. Does player control change?

Yes.

`Expand Influence` should raise local district control.

Otherwise `control` is decorative, and v0.2’s thesis weakens.

For v0.2:

```text
Expand Influence targeting a district:
+12 district control

Expand Influence targeting a venue:
+8 district control
```

Also:

```text
Run Small Job targeting a district/venue:
+3 district control

Gather Intel targeting a district/venue:
+1 district control

Lay Low:
no control gain

Bribe Official:
no control gain by default

Recruit Operative:
no control gain by default
```

Suggested implementation:

```ts
function applyDistrictControlDelta(
  state: GameState,
  order: QueuedOrder,
  resolvedEffects: PressureDelta,
): GameState {
  const districtId = resolveDistrictId(order.target);
  if (!districtId) return state;

  const deltaByAction: Partial<Record<ActionId, number>> = {
    expand_influence: order.target?.type === 'venue' ? 8 : 12,
    run_small_job: 3,
    gather_intel: 1,
  };

  const delta = deltaByAction[order.actionId] ?? 0;

  state.districts[districtId].control = clamp(state.districts[districtId].control + delta, 0, 100);

  return state;
}
```

For v0.2, district control does three things:

```text
1. Displays progress in a location.
2. Slightly improves future actions in that district.
3. Supports reports/tuning.
```

Tiny modifier:

```ts
if (districtState.control >= 40) {
  riskChance -= 2;
}

if (districtState.control >= 70) {
  riskChance -= 4;
}
```

Do not make district control directly add to global Dominion yet except through existing action effects.

# 6. Default target selection

Codex’s proposal is right.

Default:

```text

```

These can be queued without target:

```text

```

## Required-target actions

Default:

```text

```

Queue button disabled until target selected:

```text

```

UI copy:

```text
Select a target to queue this order.
```

This keeps accidental target decisions out of the game.

# 7. Rival pressure thresholds

Yes.

For v0.2:

> **Rival pressure is passive only.**

It affects:

```text
weekly passive effects
event weighting
action previews/warnings
simulation reports
```

It does **not** produce:

```text
rival turns
rival plans
rival operations
rival map movement
venue takeovers
direct AI actions
```

Thresholds:

```ts
function rivalPressureTier(pressure: number) {
  if (pressure < 25) return 'watching';
  if (pressure < 50) return 'interested';
  if (pressure < 75) return 'provoked';
  return 'retaliating';
}
```

Passive effects begin at `>= 60`.

```ts
if (nyx.pressure >= 60 && state.pressures.intel < 20) {
  state.pressures.loyalty -= 3;
}

if (knox.pressure >= 60) {
  state.pressures.heat += 3;
}
```

That’s enough.

# 8. Target history / recent activity

Use a separate `recentActivity` slice, not just the event log.

The event log is player-facing history.

`recentActivity` is simulation/input context.

```ts
type RecentActivityEntry = {
  week: number;
  actionId: ActionId;
  target?: ActionTarget;
  targetTags: string[];
  rivalId?: RivalId;
  heatDelta: number;
  dominionDelta: number;
};
```

Store only recent entries needed for weighting.

At the end of each week, prune:

```ts
state.recentActivity = state.recentActivity.filter((entry) => state.week - entry.week <= 2);
```

Why separate?

Because event log entries are prose/UI artifacts, while event weighting needs compact structured facts.

You can still render recent activity in debug reports.

# 9. Resolution order update for v0.2

Lock this order:

```text
1. Player queues orders.
2. Player assigns operative.
3. Player selects target where applicable.
4. Player clicks Advance Week.
5. For each order:
   a. calculate base effects
   b. apply operative modifiers
   c. apply target modifiers
   d. calculate risk
   e. apply effects
   f. apply local district control/heat changes
   g. apply rival pressure
   h. record recentActivity
6. Apply global weekly drift.
7. Apply local district heat cooling.
8. Apply passive rival threshold effects.
9. Select weekly event using current state + recentActivity.
10. Player chooses event response.
11. Apply event effects.
12. Check win/loss.
13. Increment week.
```

That is the canonical v0.2 turn model.

```text
Is OperatorBot now the “Sane” agent?
Yes. Keep name OperatorBot. Label it Operator/Sane in reports.

Should district/venue/rival state live inside GameState or static content plus mutable overlays?
Static definitions in content. Mutable overlays in GameState.

No direct rollup in v0.2. It modifies risk, previews, and event weighting.

Yes, lightly. Positive Heat from targeted actions increases local district heat. Local heat cools by 1/week but not below base heat.

Yes. Passive effects and event weighting only. No rival turns/plans yet.
```
