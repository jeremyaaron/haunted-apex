## Core thesis

> **Power in Haunted Apex should no longer exist in a vacuum. Every operation, front, secret, and alliance should sit inside a wider political ecology.**

v0.1 proved the pressure loop.

v0.2 made places and rivals matter.

v0.3 made operatives matter.

v0.4 made secrets, debts, favors, and consequences persist.

v0.5 made contacts useful, dangerous, and socially sticky.

v0.6 made ownership matter through fronts.

v0.7 should make **institutions** matter.

The player should no longer think only:

```text
I need Heat relief.
```

They should think:

```text
Ashline can lower Heat, but every clean corridor puts me deeper in their file.

Helix Meridian can fund the next front, but they will expect compliance later.

The Velvet House can get me Dominion in Violet Ward, but Nyx will smell it.

Ghostline can give me secrets, but the secrets come back wrong.

Chrome Maw money is fast, loud, and never free.
```

The goal is not a full 4X diplomacy simulator.

The goal is narrower:

> **Introduce factions as institutional powers with standing, suspicion, obligation, and short-term accords that create useful bargains and future pressure.**

---

# 1. Design Goal

v0.7 introduces **Factions** and **Accords**.

Factions are powerful institutions in the city. They are not individuals. They are not rivals. They are structures the player must navigate.

They can provide:

```text
protection
capital
information
market access
front support
Heat relief
Dominion acceleration
rival insulation
Ledger opportunities
```

But they also create:

```text
obligations
scrutiny
political pressure
events
debt
rival entanglement
moral/Ruin pressure
```

The player should feel that dealing with factions is useful but never clean.

# 2. What v0.7 Should Prove

v0.7 should answer:

> **Can Haunted Apex make the city feel politically alive without becoming a full diplomacy game?**

Target player stories:

```text
“I won because I took Ashline’s Clean Corridor accord, but by the end they knew too much.”

“Helix funded my fronts, then their demand event hit when I was already broke.”

“The Velvet House accord gave me enough Dominion to win, but Nyx used the same network to hurt Loyalty.”

“I ignored Chrome Maw all run, then they started leaning on my courier line.”

“I used Ghostline for secrets and ended the game with enough Intel to win and enough Ruin to regret it.”
```

The player should begin reading the city as a network of institutions, not only as individual rivals and locations.

# 5. Content Philosophy

A faction is not a vendor.

A faction is an institution with appetite.

A good faction should answer:

```text
What do they want?
What can they give?
What do they notice?
What do they punish?
What kind of player strategy makes them useful?
What kind of player strategy makes them dangerous?
```

Bad faction design:

```text
Ashline Bureau: pay 800 resources for -10 Heat.
```

Good faction design:

```text
Ashline Bureau can suppress Heat through official channels, but every use increases Suspicion and creates the possibility that your whole network becomes legible to them.
```

Factions should feel like strategic bargains with institutional memory.

# 6. Factions vs Existing Entities

## Rivals

```text
Named antagonists.
They pressure the player directly.
They control venues/districts and react to intrusion.
```

## Contacts

```text
Individual outsiders.
They provide services through Trust, Leverage, Volatility, and Exposure.
```

## Fronts

```text
Owned infrastructure.
They generate weekly value and exposure.
```

## Ledger

```text
Secrets, debts, and favors.
Persistent consequence artifacts.
```

## Factions

```text
Institutional powers.
They offer accords, track standing/suspicion/obligation, and generate political events.
```

A rival may be linked to a faction.

A contact may be linked to a faction.

A front may operate in a faction’s sphere.

A Ledger entry may implicate a faction.

The faction layer ties those systems together.

# 7. Core Faction Metrics

Each active faction has three visible metrics.

```ts
type FactionState = {
  id: FactionId;

  standing: number; // 0–100
  suspicion: number; // 0–100
  obligation: number; // 0–100

  activeAccordIds: ActiveAccordId[];

  flags: Record<string, boolean | number | string>;

  recentInteractions: FactionInteraction[];
};
```

## Standing

How favorable the faction is toward the player.

High Standing:

```text
accords become cheaper or stronger
faction events become more favorable
some penalties soften
```

Low Standing:

```text
accords become unavailable
hostile events become more likely
faction pressure increases
```

## Suspicion

How closely the faction is watching the player.

High Suspicion:

```text
inspection events
Heat pressure
front scrutiny
Ledger leak events
rival/faction complications
```

Suspicion is not the same as global Heat.

Heat is citywide attention.

Suspicion is institutional attention.

## Obligation

How much the player owes the faction.

High Obligation:

```text
demand events
debt-like pressure
forced tradeoffs
costlier future accords
```

Obligation is not a hard loss condition in v0.7.

It is political debt.

# 8. Derived Faction Status

Faction status is derived, not persisted.

```ts
type FactionStatus =
  | 'hostile'
  | 'cold'
  | 'neutral'
  | 'favorable'
  | 'watching'
  | 'indebted'
  | 'entangled';
```

Suggested derivation:

```ts
function deriveFactionStatus(faction: FactionState): FactionStatus {
  if (faction.standing <= 20) return 'hostile';
  if (faction.obligation >= 70 && faction.suspicion >= 60) return 'entangled';
  if (faction.obligation >= 70) return 'indebted';
  if (faction.suspicion >= 70) return 'watching';
  if (faction.standing >= 70) return 'favorable';
  if (faction.standing <= 40) return 'cold';
  return 'neutral';
}
```

The UI should show both the label and the underlying numbers.

# 12. Accord System

An **Accord** is a short-term faction bargain.

It is not permanent diplomacy.

It is not a full alliance.

It is a time-limited deal that gives the player value while altering faction metrics.

## Accord definition

```ts
type AccordDefinition = {
  id: AccordId;
  factionId: FactionId;

  label: string;
  description: string;

  durationWeeks: number;

  requirements?: AccordRequirement[];

  cost?: PressureCost;

  immediateEffects?: PressureDelta;
  weeklyEffects?: PressureDelta;

  factionEffectsOnStart?: FactionMetricDelta;
  factionEffectsPerWeek?: FactionMetricDelta;
  factionEffectsOnExpire?: FactionMetricDelta;

  ledgerEffectsOnStart?: LedgerEffect[];

  rivalPressureEffectsOnStart?: Partial<Record<RivalId, number>>;

  tags: string[];
};
```

## Active accord state

```ts
type ActiveAccord = {
  id: ActiveAccordId;
  definitionId: AccordId;
  factionId: FactionId;

  startedWeek: number;
  remainingWeeks: number;

  source: {
    type: 'broker_accord';
  };
};
```

## Accord limit

Recommended v0.7:

```text
Maximum active accords: 2
Maximum active accords per faction: 1
```

This prevents the player from stacking too many institutional buffs.

# 13. New Action: Broker Accord

Add one new action.

```text
Broker Accord
```

This action targets a faction and an accord option.

It costs one command point.

It does not use operative assignment in v0.7.

```ts
{
  id: 'broker_accord',
  label: 'Broker Accord',
  commandCost: 1,
  resourceCost: 0,
  effects: {},
  baseRisk: 10,
  requiresTarget: true,
  allowedTargetTypes: ['faction']
}
```

## Target model

```ts
type ActionTarget =
  | { type: 'district'; id: DistrictId }
  | { type: 'venue'; id: VenueId }
  | { type: 'rival'; id: RivalId }
  | { type: 'recruit'; id: OperativeId }
  | { type: 'ledger'; entryId: LedgerEntryId; useOptionId: LedgerUseOptionId }
  | { type: 'contact'; contactId: ContactId; optionId: ContactOptionId }
  | {
      type: 'front_opportunity';
      definitionId: FrontDefinitionId;
      districtId: DistrictId;
      venueId?: VenueId;
    }
  | { type: 'front'; id: FrontId; optionId: FrontOptionId }
  | { type: 'faction'; factionId: FactionId; accordId: AccordId };
```

# 15. Faction Touch Hooks

v0.7 should lightly integrate factions into existing actions.

Do not retrofit everything.

Use a simple helper:

```ts
function applyFactionTouch(
  state: GameState,
  context: {
    actionId: ActionId;
    target?: ActionTarget;
    resolvedEffects: PressureDelta;
  },
): GameState;
```

## Rule

If an action targets a district, venue, front, contact, or Ledger entry associated with an active faction, then faction metrics may change.

Recommended first-pass rule:

```text
If resolved action increases Dominion in associated faction sphere:
  Suspicion +2

If resolved action increases Heat in associated faction sphere:
  Suspicion +3

If resolved action uses/intensifies a Ledger entry related to faction:
  Suspicion +3 or Standing -2

If action is Broker Accord:
  use explicit accord faction effects only
```

Keep this subtle.

The faction layer should not punish every click.

# 16. Active Accord Weekly Resolution

Update weekly resolution order.

Recommended:

```text
1. Resolve queued orders.
2. Apply active accord weekly effects.
3. Apply front weekly yields.
4. Apply global weekly drift.
5. Cool local district heat.
6. Apply passive rival/contact/faction threshold effects.
7. Select weekly event.
8. Resolve event choice.
9. Decrement accord durations / expire accords.
10. Check win/loss.
```

Alternative: decrement durations before event selection. Either is fine.

Recommended for clarity:

```text
An accord applies its weekly effect during each week it is active, then loses 1 remaining week after event resolution.
```

## Expiration

When an accord expires:

```text
Remove active accord.
Apply factionEffectsOnExpire if present.
Log expiration.
```

Most v0.7 accords do not need expiration penalties.

Expiration log example:

```text
Accord expired: Ashline Clean Corridor
The clean corridor closes. The cameras remember who used it.
```

# 17. Passive Faction Threshold Effects

Keep them minimal.

## Obligation threshold

If any active faction has Obligation >= 70:

```text
Faction Demand event weight increases.
```

No automatic penalty.

## Suspicion threshold

If any active faction has Suspicion >= 70:

```text
Faction Scrutiny event weight increases.
```

If Ashline Suspicion >= 80 specifically:

```text
Front Inspection / Corp Patrol Sweep event weights increase.
```

Do not add passive weekly damage yet.

The consequences should come through events.

# 24. Agent Updates

Use accords strategically.

# 26. Balance Goals

Preserve broad v0.6 tuning.

Expected:

```text
OperatorBot: 55–75% win rate
CautiousBot: low win rate, usually Dominion shortfall
RandomBot: 5–12% win rate
AggressiveBot: viable but volatile
GreedyBot: viable but swingy
```

Faction-specific targets:

```text
Average Broker Accord uses per run: 0.5–2.0
Average active accords at a time: 0–1.5
Faction events per run: 0–2
High Obligation should matter in some runs, not every run
High Suspicion should create risk, not automatic collapse
No accord should be mandatory for OperatorBot
Faction layer should offer comeback tools and strategic acceleration
```

The player should be able to ignore factions and still win sometimes, but faction play should create different run shapes.
