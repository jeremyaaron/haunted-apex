# Haunted Apex v0.7.0 — The Accords Clarifications

## 1. Accord re-use

Each accord should be **one-use-per-run**.

A faction can offer multiple accords during a run, but a specific accord cannot be brokered again after it expires.

Example:

```text
Allowed:
Week 2: Ashline — Clean Corridor
Week 5: Ashline — Inspection Delay

Not allowed:
Week 2: Ashline — Clean Corridor
Week 5: Ashline — Clean Corridor again
```

Reasoning:

```text
Prevents repetitive optimal loops.
Makes each accord feel like a meaningful bargain.
Keeps the 8-week campaign from becoming “spam Clean Corridor.”
Makes simulation reports easier to interpret.
```

Implementation:

```ts
type FactionState = {
  id: FactionId;
  standing: number;
  suspicion: number;
  obligation: number;
  usedAccordIds: AccordId[];
  activeAccordIds: ActiveAccordId[];
  // ...
};
```

Lock:

```text
Accord definitions are one-use-per-run.
Different accords from the same faction may still be brokered later.
```

## 2. Front-specific accord hooks

Implement **one concrete showcase hook** and defer the more invasive one.

### Implement now: Ashline — Inspection Delay

Yes:

```text
Ashline Inspection Delay:
Reduce highest-exposure owned Front by 10.
```

This is straightforward, visible, and reinforces Ashline’s identity as institutional Heat/front-control.

Preview should show:

```text
Highest-exposure front: Zero Mercy Cut
Zero Mercy Cut Exposure -10
```

If no owned front exists somehow, disable this accord or show no valid target. Since The Pale Circuit always exists, this should normally be valid.

### Defer for now: Helix — Permit Shell cost discount

Do **not** implement the `-200 establish/upgrade Front cost while active` hook in v0.7 unless Codex already has an easy generic cost-modifier pipeline.

Reasoning:

```text
Cost discounts require preview and resolution changes across Invest in Front.
They introduce timing questions.
They increase the chance that preview and resolution diverge.
```

For v0.7, keep Helix `Permit Shell` as general pressure/resource effects.

Example:

```text
Immediate: Heat -2
Weekly: Resources +150
Faction: Standing +4, Suspicion +5, Obligation +8
```

Future hook:

```ts
activeAccordModifiers.frontCostDiscount = 200;
```

but not required for v0.7.

Lock:

```text
Ashline Inspection Delay gets a concrete front exposure reduction.
Helix Permit Shell does not modify front costs in v0.7.
```

## 3. Faction touch strength

Limit faction touch in v0.7.

Do **not** apply faction touch to every normal district/venue action yet.

Faction metrics should change from:

```text
Broker Accord explicit effects
Front actions in associated faction sphere
Contact actions with associatedFactionId
Ledger use/event outcomes with relatedFactionId
Faction events
```

Do not broadly apply faction touch to:

```text
Gather Intel at any associated venue
Run Small Job in any associated district
Expand Influence anywhere linked to a faction
Bribe Official generally
```

unless the action has an explicit faction-related target/effect.

This keeps the tuning surface manageable.

Locked v0.7 faction touch sources:

```text
1. Broker Accord — explicit accord effects.
2. Front establish/upgrade in faction-associated district/venue/tag.
3. Manage Contact when contact has associatedFactionId.
4. Work the Ledger when Ledger entry has relatedFactionId and declared faction effects.
5. Faction events.
```

No broad “all normal actions in faction space alter faction state” yet.

## 4. Faction event copy

Use **generic mechanics with light faction-specific labels/flavor**.

Do not write fully bespoke mechanics and choices per faction in v0.7.

Example:

```text
Faction Demand: Helix Meridian
Faction Scrutiny: Ashline Bureau
Market Access: Velvet House
```

The event can use the same choice structure, but the title and intro text should include faction name/archetype.

Good v0.7 approach:

```ts
type FactionEventDefinition = {
  id: EventId;
  template: 'faction_demand' | 'faction_scrutiny' | 'market_access';
  eligibleFactionTags?: FactionRoleTag[];
  titleTemplate: string;
  bodyTemplate: string;
  choices: EventChoiceDefinition[];
};
```

With light flavor substitutions:

```text
Ashline Bureau → files, corridors, patrols, compliance
Helix Meridian → permits, capital, boardrooms, contracts
Velvet House → invitations, rooms, guest lists
Chrome Maw → docks, muscle, shipments
Ghostline Communion → signals, dead channels, strange markets
```

Lock:

```text
Faction events use generic mechanics and light faction-flavored copy.
No bespoke faction event trees in v0.7.
```

## 7. Associated entity links

Use both patterns, but for different purposes.

## Direct fields on entity definitions

Add direct association fields where the entity naturally belongs to one faction.

```ts
type RivalDefinition = {
  // existing...
  associatedFactionId?: FactionId;
};

type ContactDefinition = {
  // existing...
  associatedFactionId?: FactionId;
};
```

Examples:

```text
Nyx Ardent → Velvet House
Knox Marrow → Chrome Maw
Captain Hollis → Ashline Bureau
Veyra Lux → Velvet House
Ciro Moth → Ghostline Communion
```

## FactionDefinition lists/tags

Keep faction-side associations for districts, venues, and front tags.

```ts
type FactionDefinition = {
  associatedDistrictIds?: DistrictId[];
  associatedVenueIds?: VenueId[];
  associatedFrontTags?: FrontRoleTag[];
  associatedRivalIds?: RivalId[];
  associatedContactIds?: ContactId[];
};
```

Reasoning:

```text
Rivals/contacts often have a clear institutional alignment.
Districts/venues/fronts are better represented as faction sphere-of-influence lists.
```

Lock:

```text
Use direct associatedFactionId on rivals and contacts.
Use faction-side associations for districts, venues, and front tags.
```

## 8. Weekly timing

Lock the timing as:

> **Brokered accords apply immediate effects this week, but weekly effects begin on the next Advance Week resolution.**

This avoids double-dipping and makes accord duration easier to reason about.

## Recommended model

When the player brokers an accord in Week N:

```text
During Week N resolution:
- Pay cost
- Apply immediate effects
- Apply factionEffectsOnStart
- Create ActiveAccord
- Do not apply weeklyEffects yet
- Do not decrement duration yet
```

On Week N+1 resolution:

```text
- Apply weeklyEffects
- Apply factionEffectsPerWeek
- Decrement remainingWeeks
```

After the final weekly tick:

```text
- Expire accord
- Apply factionEffectsOnExpire, if any
- Log expiration
```

Implementation shape:

```ts
type ActiveAccord = {
  id: ActiveAccordId;
  definitionId: AccordId;
  factionId: FactionId;
  startedWeek: number;
  remainingWeeks: number;
  firstWeeklyEffectWeek: number;
};
```

When created:

```ts
firstWeeklyEffectWeek = state.week + 1;
remainingWeeks = definition.durationWeeks;
```

Weekly application condition:

```ts
state.week >= activeAccord.firstWeeklyEffectWeek;
```

Lock:

```text
Immediate effects apply on broker week.
Weekly effects begin the following week.
Duration counts weekly-effect ticks, not total calendar weeks.
```

This should be clearly shown in preview:

```text
Immediate:
Heat -6

Starting next week for 2 weeks:
Heat -3/week
```

## 9. RandomBot target

Treat the old `5–12%` RandomBot win rate as aspirational, not mandatory.

Given v0.6 accepted RandomBot at 0%, v0.7 should not be blocked because RandomBot does not win.

RandomBot’s main purpose now is:

```text
valid action coverage
crash/lockup detection
weird-state discovery
rules validation
```

Updated target:

```text
RandomBot:
0–10% win rate acceptable.
Must not crash, softlock, or produce invalid state.
```

The main balance indicators remain:

```text
OperatorBot remains viable.
CautiousBot usually fails by Dominion shortfall.
AggressiveBot can win but overcreates Heat/risk.
GreedyBot can win but is swingy and obligation-prone.
```

Lock:

```text
RandomBot win rate is diagnostic, not a core balance target.
```
