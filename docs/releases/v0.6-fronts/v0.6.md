## Core thesis

> **Power should stop being only influence and start becoming infrastructure.**

v0.1 proved the pressure loop.

v0.2 made places and rivals matter.

v0.3 made operatives matter.

v0.4 made secrets, debts, and favors persist.

v0.5 made contacts socially useful and dangerous.

v0.6 should make **ownership** matter.

The player should no longer think only:

```text
I have 48 Dominion.
```

They should think:

```text
I own The Pale Circuit.
The courier line is paying out, but its exposure is getting dangerous.
The black clinic is keeping my crew alive.
Zero Mercy would make me rich, but Knox will notice.
I can push for immediate Dominion, or build a front that pays off later.
```

The goal is not a full economy simulator. It is narrower:

> **Introduce owned fronts as persistent assets that generate weekly value while creating exposure, rival attention, and event hooks.**

---

# 1. Design Goal

v0.6 introduces **Fronts**: semi-legitimate or illegitimate businesses that the player owns or controls.

Fronts are persistent assets attached to districts or venues.

They produce recurring value:

```text
Resources
Intel
Loyalty
Heat reduction
Dominion
district control
Ruin manipulation
```

But they also create recurring risk:

```text
Exposure
rival pressure
inspections
staff problems
debt hooks
Ledger events
contact complications
```

Fronts should make the player consider time horizon.

The current game mostly asks:

```text
What do I need this week?
```

Fronts add:

```text
What am I building that will still matter three weeks from now?
```

---

# 2. What v0.6 Should Prove

v0.6 should answer:

> **Can Haunted Apex make investment and ownership strategically interesting inside a short 8-week run?**

Target player stories:

```text
“I won because I built the courier line early and it funded my final push.”

“The Pale Circuit kept my loyalty alive, but I ignored its exposure and got inspected.”

“I invested in the black clinic because Juno was breaking, and it quietly saved the run.”

“I established a fight pit in Knox territory and spent the rest of the game paying for that attention.”

“I had enough cash from fronts to win, but the businesses made me too visible.”
```

The player should feel that fronts are not passive upgrades. They are assets with heat signatures.

# 5. Content Philosophy

A front is not an upgrade button.

A front is a **persistent bargain**.

It should answer:

```text
What does this produce?
What does this hide?
Who notices it?
How quickly does it pay back?
What kind of trouble does it attract?
What kind of strategy does it support?
```

Bad front design:

```text
Nightclub: +500 Resources per week.
```

Good front design:

```text
The Pale Circuit provides Loyalty and light income, but if its exposure rises, it becomes a place where rivals, contacts, and officials can reach you.
```

Fronts should feel like:

```text
assets
cover stories
liabilities
social anchors
pressure converters
```

# 6. Fronts vs Existing Entities

## Districts

```text
Places where power is contested.
Have local heat/control/rival context.
```

## Venues

```text
Named places that modify action rewards/risks.
May be controlled by rivals.
```

## Fronts

```text
Owned or controlled infrastructure.
Generate recurring value.
Accumulate exposure.
Can be invested in, upgraded, cooled, inspected, or compromised.
```

A venue can inspire or host a front, but a front is a player-owned asset.

Example:

```text
Venue: The Glass Saint
Rival-controlled elite memory lounge.

Front: Private Salon
Player-controlled social/intel business established in Violet Ward.
```

# 7. Core Front Metrics

Each front has two main mutable metrics in v0.6:

```text
Level
Exposure
```

## Level

How developed the front is.

v0.6 supports:

```text
Level 1
Level 2
```

Do not add Level 3 yet.

Level affects:

```text
weekly yield
front event intensity
exposure gain
strategic value
```

## Exposure

How visible, suspicious, or compromised the front has become.

Exposure affects:

```text
front event eligibility
Heat risk
rival attention
inspection events
Ledger debt/secret hooks
```

Exposure is not global Heat, but it can create Heat.

# 8. Front Status

Front status is derived from Exposure.

```ts
type FrontStatus = 'quiet' | 'noticed' | 'hot' | 'compromised';
```

Suggested derivation:

```ts
function deriveFrontStatus(exposure: number): FrontStatus {
  if (exposure < 30) return 'quiet';
  if (exposure < 60) return 'noticed';
  if (exposure < 80) return 'hot';
  return 'compromised';
}
```

## Status meaning

```text
Quiet:
Safe. Normal yields.

Noticed:
Event weights begin increasing.

Hot:
Inspection/rival pressure events become likely.

Compromised:
Front can still operate, but front events become severe.
```

Do not automatically shut down compromised fronts in v0.6.

Let the player decide whether to cool them, exploit them, or ride the danger.

# 12. New Action: Invest in Front

Add one new action.

```text
Invest in Front
```

This action handles both:

```text
Establish new front
Upgrade owned front
```

It costs one command point.

It does not require operative assignment.

```ts
{
  id: 'invest_front',
  label: 'Invest in Front',
  commandCost: 1,
  resourceCost: 0,
  effects: {},
  baseRisk: 10,
  requiresTarget: true,
  allowedTargetTypes: ['front_opportunity', 'front']
}
```

## Target model

```ts
type ActionTarget =
  | { type: 'district'; id: DistrictId }
  | { type: 'venue'; id: VenueId }
  | { type: 'rival'; id: RivalId }
  | { type: 'recruit'; id: OperativeId }
  | { type: 'ledger'; entryId: LedgerEntryId; useOptionId: LedgerUseOptionId }
  | { type: 'contact'; contactId: ContactId; optionId: ContactOptionId }
  | {
      type: 'front_opportunity';
      definitionId: FrontDefinitionId;
      districtId: DistrictId;
      venueId?: VenueId;
    }
  | { type: 'front'; id: FrontId; optionId: FrontOptionId };
```

## Front option IDs

For v0.6:

```ts
type FrontOptionId = 'upgrade' | 'cool';
```

But `cool` should usually be handled by Lay Low, not Invest in Front.

Recommended v0.6:

```text
Invest in Front:
- establish opportunity
- upgrade owned front

Lay Low:
- can target owned front to reduce exposure
```

# 15. Weekly Front Yield

Fronts produce value during weekly resolution.

Recommended order:

```text
1. Resolve queued orders.
2. Apply front weekly yields.
3. Apply global weekly drift.
4. Cool local district heat.
5. Apply passive rival/contact effects.
6. Select weekly event.
7. Resolve event choice.
8. Check win/loss.
```

## Yield calculation

```ts
function calculateFrontWeeklyYield(front, definition): PressureDelta {
  let yield = { ...definition.baseWeeklyYield };

  if (front.level >= 2) {
    yield = addPressureDelta(yield, definition.level2BonusYield);
  }

  return yield;
}
```

## Exposure gain

Each active front gains exposure weekly.

```ts
front.exposure += definition.exposurePerWeek;

if (front.level === 2) {
  front.exposure += 1;
}
```

Keep it simple.

## District control yield

If definition has `districtControlYield`, apply it weekly:

```ts
district.control += definition.districtControlYield * front.level;
```

Clamp district control.

# 20. Rival Integration

If a front is established in a rival-controlled district or venue, the controlling rival gains pressure.

```ts
function applyFrontRivalPressureOnEstablish(front, definition, state) {
  const rival = getControllingRivalForDistrictOrVenue(front.districtId, front.venueId);

  if (!rival) return;

  const gain = definition.rivalPressureOnEstablish ?? 6;
  rival.pressure += gain;
}
```

If a front has `rivalPressurePerWeek`, apply it weekly.

This makes risky fronts like Zero Mercy Cut feel dangerous.

# 22. Agent Updates

Agents should become front-aware.

## OperatorBot

Should:

```text
Invest early in one useful front if resources allow.
Prefer fronts that solve current strategic gaps.
Avoid high-exposure fronts if Heat is already high.
Target Lay Low at hot fronts when exposure >= 70 and the run has time.
Upgrade fronts only if payback horizon is reasonable.
```

Possible rules:

```text
If Week <= 3 and Resources >= setupCost + 1000, establish best front.
If Heat >= 70, prefer Shell Gallery / Black Clinic / Courier Line.
If Resources <= 1500 and Week <= 5, prefer Courier Line / Zero Mercy Cut.
If front exposure >= 75, consider Lay Low targeting front.
Do not upgrade after Week 6 unless upgrade gives immediate Dominion/Heat relief.
```

## CautiousBot

```text
Prefers Shell Gallery, Black Clinic, Courier Line.
Avoids Zero Mercy Cut.
Cools fronts early.
Likely still misses Dominion target.
```

## AggressiveBot

```text
Prefers Zero Mercy Cut and Dominion-producing fronts.
Upgrades early if possible.
Ignores exposure until severe.
```

## GreedyBot

```text
Prefers highest resource yield.
Establishes/Upgrades money fronts.
Accepts exposure and debt events.
```

## RandomBot

Random valid front investment/cooling.

# 26. Technical Guidance

## Static definitions, mutable state

Keep the established pattern.

```text
FrontDefinition = static content
FrontState = mutable GameState
FrontOpportunity = seeded run setup
```

## Suggested modules

```text
content/fronts.ts
content/frontEvents.ts

engine/fronts/generateFrontOpportunities.ts
engine/fronts/frontSelectors.ts
engine/fronts/deriveFrontStatus.ts
engine/fronts/resolveInvestFront.ts
engine/fronts/applyFrontWeeklyYields.ts
engine/fronts/applyFrontExposure.ts
engine/fronts/applyFrontEffects.ts
engine/fronts/frontEventEligibility.ts
engine/fronts/frontReports.ts
```

## Avoid hardcoding front IDs in engine logic

Bad:

```ts
if (front.definitionId === 'front_zero_mercy_cut') {
  state.pressures.heat += 4;
}
```

Better:

```ts
applyPressureDelta(state, frontDefinition.baseWeeklyYield);
applyFrontExposure(state, frontDefinition.exposurePerWeek);
```

Hand-authored event definitions are fine; generic engine logic should read definitions.
