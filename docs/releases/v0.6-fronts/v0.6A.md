# Haunted Apex v0.6.0 — Fronts Clarifications

## 1. Does The Pale Circuit count against the cap?

Yes.

```text
Owned front cap: 3
Starting owned front: The Pale Circuit
Available additional front slots: 2
```

That is intentional. The cap should force strategic choices. If the player could own The Pale Circuit plus three more fronts, the system would start feeling like a checklist.

Locked rule:

```text
The Pale Circuit counts as one of the player’s three owned fronts.
```

---

## 2. Do front opportunities stay fixed for the run?

Yes, for v0.6.

Front opportunities are seeded at new-game generation and remain visible/available unless established.

No mid-run reveal system yet.

Locked rule:

```text
Front opportunities are fixed seeded run content in v0.6.
```

Later versions can add:

```text
events reveal special front opportunities
contacts unlock front opportunities
Ledger secrets reveal hidden fronts
rivals lose control of venues
```

But not now.

---

## 3. Can the player own only one instance of each front definition?

Yes.

For v0.6:

```text
One instance per FrontDefinition.
```

So the player cannot own:

```text
Courier Line #1
Courier Line #2
Courier Line #3
```

That would turn the system into a stacking economy too early.

Locked rule:

```text
A front definition can be owned at most once per run.
```

---

## 4. The Pale Circuit as venue vs front

Yes — use the same displayed name but separate IDs.

This means:

```text
venue_pale_circuit = the location
front_pale_circuit = the player-controlled operation inside that location
```

That is clean and future-proof.

Player-facing copy can simply say:

```text
The Pale Circuit
Violet Ward · Starting Front
```

Internally:

```ts
venueId: 'venue_pale_circuit';
frontId: 'front_pale_circuit';
definitionId: 'front_pale_circuit';
```

Locked rule:

```text
The Pale Circuit front is a player-owned operation associated with the existing Pale Circuit venue.
```

---

## 5. Rival pressure on fronts: inferred or stored?

Store the related rival on the **front opportunity** at generation time, but derive it initially from district/venue.

Why:

```text
The UI preview and reports need a stable, explicit “related rival.”
Future versions may allow ownership/control changes.
Generation-time capture makes the opportunity self-describing.
```

Recommended shape:

```ts
type FrontOpportunity = {
  id: FrontOpportunityId;
  definitionId: FrontDefinitionId;
  districtId: DistrictId;
  venueId?: VenueId;
  relatedRivalId?: RivalId;
};
```

When generating the opportunity:

```ts
relatedRivalId = getControllingRivalForDistrictOrVenue(districtId, venueId);
```

When established, copy it into `FrontState`:

```ts
type FrontState = {
  id: FrontId;
  definitionId: FrontDefinitionId;
  districtId: DistrictId;
  venueId?: VenueId;
  relatedRivalId?: RivalId;
  level: 1 | 2;
  exposure: number;
  // ...
};
```

Locked rule:

```text
The related rival is inferred at opportunity generation and stored on the opportunity/front state.
```

---

## 6. Does exposure have a passive direct penalty?

No, not by default.

Exposure affects:

```text
front status
event eligibility
event severity
preview warnings
reports
```

Exposure does **not** automatically apply global Heat, Loyalty damage, Resource drain, etc.

Exception:

```text
If a FrontDefinition explicitly defines weekly Heat, Ruin, or rival pressure, apply that.
```

Example:

```text
Zero Mercy Cut weekly yield includes Heat +1.
Zero Mercy Cut may define rivalPressurePerWeek +2.
```

But generic exposure itself is not a passive damage aura.

Locked rule:

```text
Exposure is primarily event-driven in v0.6.
```

This keeps it legible:

```text
High exposure means “this front is likely to cause trouble,” not “numbers mysteriously decay every week.”
```

## 7. Front event target selection

Use weighted-random among eligible fronts, biased toward exposure.

Do **not** always pick the highest-exposure front.

Why:

```text
Highest-only becomes predictable.
Pure random ignores accumulated risk.
Weighted-random gives danger texture while preserving causality.
```

Recommended weighting:

```ts
weight = 10 + front.exposure;
```

Then add bonuses:

```text
+20 if exposure >= 80
+10 if front is in rival territory and event is rival-related
+10 if front level is 2
+5 if front was established/upgraded recently
```

Locked rule:

```text
Front events select a weighted-random eligible front, with exposure as the main weight.
```

Player-facing, high-exposure fronts should still feel more dangerous because they get selected more often.

## 8. New Ledger definitions for front events?

Mostly reuse existing Ledger entries.

Add only one or two new entries if needed.

Recommended v0.6 additions:

```text
Secret: Back Room Guest List
Debt: Dirty Books
```

That is enough.

Everything else can reuse existing:

```text
Favor: Hidden Route
Secret: Dead Channel Trace
Debt: Contaminated Money
Debt: Fabricated Evidence
Debt: Unfunded Promise
```

Locked rule:

```text
Reuse existing Ledger definitions where semantically acceptable. Add at most two front-specific Ledger entries in v0.6.
```

Suggested new entries:

## Secret — Back Room Guest List

```text
A list of names from a private room that officially has no door.
Use: Intel/Dominion leverage, or Heat reduction through blackmail.
```

## Debt — Dirty Books

```text
Accounting irregularities useful today and poisonous tomorrow.
Settle: pay Resources or spend Intel to bury the trail.
```

## 9. Contact integration: real mechanics or future-ready tags?

Keep it future-ready for v0.6, with maybe one showcase integration only if trivial.

Given the system size, I would **defer full mechanical contact modifiers**.

Minimum v0.6:

```text
Front definitions include role tags.
Front opportunities can display related contact hints if easy.
Front event copy may reference active contacts when appropriate.
No core contact modifier system required.
```

Optional showcase, only if easy:

```text
If Dr. Mercy is active, Black Clinic setup cost -200 or Black Clinic event weight slightly reduced.
```

But I would not make this a required v0.6 acceptance criterion.

Locked rule:

```text
No required mechanical contact modifiers in v0.6. Keep contact integration mostly narrative/future-ready.
```

The game already has a lot of moving parts. Fronts need to stand on their own first.

---

## 10. UI placement

Use a **tabbed/shared side panel** rather than adding another full always-visible panel.

The UI is already dense. Fronts need room, but not at the cost of making the command screen unreadable.

Recommended layout:

```text
Main command area remains focused on:
- Actions
- Selected action preview
- Queued orders
- Advance Week

Secondary intelligence panel uses tabs:
- Operatives
- Rivals
- Ledger
- Contacts
- Fronts
- Log / Debug
```

Or, if the current app already has panels:

```text
Add Fronts as a new tab beside Ledger/Contacts.
```

The Fronts tab should have:

```text
Owned Fronts
Front Opportunities
Front details on selection
```

Do not create a separate route/page yet.

Locked rule:

```text
Fronts should live in a tabbed/shared panel to control page height.
```

# Locked v0.6 Rules Block for Codex

```text
The Pale Circuit counts against the cap of 3 owned fronts.

The player can establish at most two additional fronts in v0.6.

Front opportunities are seeded once at new-game creation and remain fixed for the run.

The player may own only one instance of each FrontDefinition.

The Pale Circuit front shares the displayed name with the Pale Circuit venue but uses a separate front ID.

The front represents the player-owned operation inside the venue.

Front opportunities store relatedRivalId at generation time, derived from district/venue control.

Established FrontState also stores relatedRivalId.

Exposure has no generic passive penalty.

Exposure affects status, event eligibility, event severity, warnings, and reports.

Only explicit front definition yields/effects apply weekly pressure changes.

Front events select a weighted-random eligible front, biased by exposure.

Reuse existing Ledger definitions where possible.

Add at most two new front-specific Ledger entries: Back Room Guest List and Dirty Books.

Do not implement required mechanical contact modifiers in v0.6.

Contact integration is future-ready through tags, copy, and optional non-critical hooks.

Fronts UI should be a tabbed/shared panel, not a new full-page layout.
```

# Additional implementation notes

## Front cap behavior

If owned front count is already 3:

```text
Establish actions disabled.
Upgrade actions still allowed.
```

Do not make the player abandon/sell fronts yet.

## Opportunity visibility

Even if front cap is reached, opportunities can remain visible but disabled:

```text
Front cap reached.
```

That helps the player understand the missed options.

## Upgrade visibility

Owned fronts at level 2 should show:

```text
Max Level
```

and disable upgrade.

## Exposure preview

Every front investment preview should show:

```text
Starting Exposure
Weekly Exposure Gain
Status after investment
```

Example:

```text
Exposure:
Starts at 20
+5/week
Initial Status: Quiet
```

For an upgrade:

```text
Exposure:
+12 immediately
+1 additional weekly pressure from Level 2
```

## Front event copy

Always include the selected front name.

Good:

```text
Front Inspection: Zero Mercy Cut
```

Bad:

```text
Front Inspection
```

Specificity is what makes the system feel alive.

---

# Final design stance

v0.6 should make the player think:

```text
I need recurring value, but anything I own can be found.
```

That is the core of **Fronts**.

The system should add ownership, payback timing, and exposure management — not a full business sim.
