## Core thesis

> **A player should not need to become a Haunted Apex grandmaster to win a normal run. Standard play should be winnable through clear current-state reasoning, not deep search.**

v0.1 proved the pressure loop.

v0.2 made territory and rivals matter.

v0.3 made operatives matter.

v0.4 made secrets, debts, favors, and consequences persist.

v0.5 made contacts useful, dangerous, and socially sticky.

v0.6 made ownership matter through fronts.

v0.7 made institutions matter through factions and accords.

v0.8 gave each run a campaign identity.

v0.9 should make the game **learnable and reliably winnable** by introducing:

```text
a strong current-state HandlerBot
a visible Advisor UX
training-wheel guidance levels
validated run support
better new-player clarity
```

The goal is not to create a chess engine.

The goal is:

> **A competent policy bot, using only current state and legal previews, should be able to win reliably. The Advisor should expose that reasoning to the player.**

---

# 1. Design Goal

v0.9 introduces **HandlerBot**, a new high-quality current-state strategy bot, and uses it to power the player-facing **Advisor**.

HandlerBot is not OperatorBot.

OperatorBot is a sane benchmark strategy.

HandlerBot is a teaching and validation strategy.

The difference:

```text
OperatorBot:
“How does a reasonably sane baseline strategy perform?”

HandlerBot:
“What should a player do right now if they want a reliable win?”
```

HandlerBot must not require a search tree or deep future-state calculation.

It should:

```text
look at current GameState
evaluate legal current-turn options
choose up to 2 orders
choose event responses
repeat for 8 weeks
win reliably
produce explanations
```

If HandlerBot cannot win reliably, the first assumption should be:

```text
The game is too hard, too swingy, too opaque, or incorrectly tuned.
```

Not:

```text
The bot needs to become a chess engine.
```

# 2. What v0.9 Should Prove

v0.9 should answer:

> **Can Haunted Apex teach a new player how to win without dumbing the game down?**

Target player stories:

```text
“I followed the Handler for my first run and finally understood the loop.”

“The advice told me exactly what to do, but I still clicked everything myself, so I felt like I played.”

“After a few wins with Handler guidance, I switched to Coach mode and started making my own calls.”

“I lost after turning the Advisor off, but I understood why.”

“The game is complex, but it no longer feels like I’m being thrown into a cockpit without labels.”
```

The player should be able to graduate naturally:

```text
Handler → Coach → Hints → Off
```

# 5. Design Contract

The core v0.9 design contract is:

> **Standard Haunted Apex should be winnable by robust current-state play. Deep lookahead may improve score or style, but should not be required for victory.**

A player should be able to win by learning principles like:

```text
Stay on Dominion pace.
Do not let Heat enter crisis.
Do not let Loyalty collapse.
Keep enough Resources for recovery.
Use Secrets and Favors when they solve current problems.
Do not overuse stressed operatives.
Do not ignore Front Exposure, Faction Obligation, Contact Volatility, or Rival Pressure when they are red.
```

If winning requires:

```text
predicting exact future event sequences
threading a narrow hidden route
knowing that one obscure early bargain is mandatory
searching an 8-week tree of possible outcomes
```

then Standard is too hard or too opaque.

# 7. HandlerBot Philosophy

HandlerBot is not an oracle.

HandlerBot should not:

```text
perform multi-week search
inspect future event rolls
solve exact future branches
play hidden perfect-information chess
```

HandlerBot may:

```text
evaluate all legal current-turn action combinations
use current legal previews
use visible state
use known rules
use campaign tension information
use current event choices
use current run history
use status bands and danger thresholds
```

This is important.

HandlerBot can be smart, but its reasoning should be explainable to a player.

# 8. HandlerBot Turn Policy

Every command phase, HandlerBot should follow this priority hierarchy.

## Priority 1 — Prevent immediate loss

If any primary loss pressure is in crisis, respond first.

Crisis examples:

```text
Heat dangerously close to 100
Loyalty dangerously close to 0
Resources close to going negative
```

HandlerBot should prioritize:

```text
Heat reducers
Loyalty stabilizers
Resource recovery
event choices that prevent collapse
```

## Priority 2 — Stay on Dominion pace

Calculate required Dominion pace.

```ts
dominionNeeded = dominionTarget - currentDominion;
weeksRemaining = maxWeeks - currentWeek + 1;
dominionPerWeekNeeded = dominionNeeded / weeksRemaining;
```

If behind pace:

```text
choose at least one Dominion-producing action if not in crisis
```

If far behind:

```text
accept controlled risk for Dominion
```

If ahead of pace:

```text
stabilize pressures and prepare recovery tools
```

## Priority 3 — Use available leverage

If the player has an active Ledger entry, contact service, accord, front option, or favor that solves a current pressure problem, HandlerBot should use it.

Examples:

```text
Use a Heat-reducing Secret when Heat is high.
Settle a dangerous Debt if resources allow.
Call a Favor if it prevents crisis.
Broker an Ashline accord if Heat is dangerous.
Use a contact service if Loyalty or Heat needs stabilization.
```

## Priority 4 — Manage secondary red zones

Secondary systems should not dominate every week, but HandlerBot must not ignore red-zone warnings.

Examples:

```text
Operative Stress >= Breaking
Front Exposure >= Hot/Compromised
Faction Obligation >= High
Faction Suspicion >= High
Contact Volatility >= High
Rival Pressure >= Retaliating
```

HandlerBot should stabilize these when primary pressures are safe.

## Priority 5 — Build future value only when there is time

HandlerBot should invest in fronts, accords, contacts, and setup actions early enough to matter.

Examples:

```text
Early run:
fronts and recurring value can be good.

Late run:
only immediate effects matter unless the front/accord directly helps win.
```

## Priority 6 — Avoid unnecessary ugliness

HandlerBot should not take Debt/Ruin/Obligation/Exposure unless it solves a meaningful current problem.

This keeps HandlerBot teachable.

# 9. HandlerBot Candidate Evaluation

HandlerBot can enumerate legal current-turn candidates.

This is not deep search.

It is current-turn tactical evaluation.

## Candidate generation

```text
Generate all legal combinations of up to 2 orders.
Respect command points.
Respect target requirements.
Respect operative availability.
Respect resource affordability.
Respect roster/front/accord/contact/faction constraints.
```

## Candidate scoring

Each candidate is scored after applying current-turn previews/resolution expectations.

Suggested scoring categories:

```text
loss prevention
Dominion pace improvement
Heat safety
Loyalty safety
Resource safety
available recovery assets
secondary risk control
campaign-specific priorities
cost / Debt / Ruin / Obligation avoidance
```

## Important

The score must prefer **robust states** over flashy states.

Bad Handler behavior:

```text
maximizes Dominion but leaves Heat at 98
```

Good Handler behavior:

```text
gets enough Dominion while preserving a recovery path
```

# 10. HandlerBot Event Policy

HandlerBot must advise event choices too.

This is essential.

If the Advisor tells the player what orders to queue but abandons them during event decisions, new players will still lose and feel tricked.

## Event response priorities

```text
1. Prevent immediate loss.
2. Preserve Dominion pace if safe.
3. Reduce red-zone pressure.
4. Avoid unaffordable costs.
5. Avoid severe Debt/Ruin/Obligation unless necessary.
6. Prefer choices that preserve future recovery tools.
7. If no crisis, choose the best strategic gain.
```

## Event recommendation output

```ts
type HandlerEventRecommendation = {
  eventId: EventId;
  choiceId: EventChoiceId;
  confidence: AdvisorConfidence;
  reason: string;
  warnings: string[];
};
```

Example:

```text
Recommended Choice:
Pay for clean passage.

Reason:
Heat is the most dangerous pressure this week, and you can afford the Resource cost.
```

# 11. Advisor Modes

Add player-facing Advisor modes.

```ts
type AdvisorMode = 'off' | 'hints' | 'coach' | 'handler';
```

## Off

No Advisor panel.

Tooltips and Field Guide remain available.

## Hints

Shows current-state warnings and opportunities only.

No exact commands.

Example:

```text
You are behind Dominion pace.
Heat is stable.
You have a useful Secret in the Black Ledger.
One operative is close to Unstable stress.
```

## Coach

Gives strategic suggestions, but not full exact orders.

Example:

```text
You need Dominion this week. Expand Influence is likely correct.
Use a social operative if possible.
Avoid Zero Mercy unless you can absorb the Heat.
```

## Handler

Gives exact recommendations.

Example:

```text
Recommended Order 1:
Expand Influence
Target: Violet Ward
Operative: Saint Calder

Recommended Order 2:
Gather Intel
Target: Ghostline Market
Operative: Mara Voss

Reason:
You are behind Dominion pace, and Heat is stable enough to take a controlled risk while building Intel.
```

Handler also recommends event choices.

---

# 12. No Auto-Apply in v0.9

v0.9 should **not** auto-queue or auto-apply recommendations.

The player must still:

```text
select action
select target
select operative where applicable
queue order
advance week
choose event response
```

This matters because:

```text
the player learns the interface
the player feels agency
the player feels they played the game
recommendations become instruction, not autopilot
```

Handler mode can highlight recommended choices, but it should not execute them.

Locked rule:

```text
No Apply Recommended Orders button in v0.9.
```

# 15. Handler Validation

v0.9 should introduce Handler validation.

This is not deep solvability proof. It is current-state policy validation.

## Validation goal

```text
Can HandlerBot win this generated run by following its own current-state policy?
```

## HandlerValidationResult

```ts
type HandlerValidationResult = {
  status: 'handler_win' | 'handler_loss' | 'invalid_state';

  seed: string;
  campaignTensionId: CampaignTensionId;
  runMode: RunMode;

  resultWeek?: number;
  finalPressures?: Pressures;
  lossCause?: string;

  decisionTrace: HandlerDecisionTraceEntry[];
};
```

## Decision trace

```ts
type HandlerDecisionTraceEntry = {
  week: number;
  phase: 'command' | 'event';
  recommendationSummary: string;
  chosenOrders?: HandlerRecommendedOrder[];
  chosenEventChoiceId?: EventChoiceId;
  reason: string;
};
```

This trace becomes a development artifact and possibly a future Advisor teaching source.

# 17. Runtime Validation vs Harness Validation

Do not overcomplicate browser startup.

## Minimum v0.9

Harness validation:

```text

Reports show Handler win rate by campaign and run mode.
```

## Preferred v0.9 if cheap

Runtime validation for Training Run:

```text
Training run uses the fixed Training config.
HandlerBot must win that config.
```

## Optional later

Runtime validation for random Standard runs:

```text
generate candidate run
simulate HandlerBot
if Handler wins, accept
if Handler loses, reroll or mark unvalidated
```

Since HandlerBot is not doing deep search, this may be feasible. But it is not mandatory for the first implementation if the harness does the validation.

v0.9 does not require runtime run certification or a certified seed bank. The validation gate is
build-level confidence from deterministic Handler trials.

# 19. Tuning Policy

v0.9 should introduce a clear rule:

> **If HandlerBot cannot win Standard with current-state reasoning, the game should be tuned toward legibility and reliability.**

Handler losses should be classified.

## Handler loss categories

```text
Dominion shortfall
Heat collapse
Loyalty collapse
Resource collapse
Event spike
Campaign overtuning
Bad opening generation
Handler policy weakness
Invalid state / bug
```

## Fix preference order

1. Fix bugs or invalid state.
2. Fix Handler blind spots if obvious and teachable.
3. Improve previews/guidance if the issue is legibility.
4. Tune campaign modifiers.
5. Tune event severity/weights.
6. Tune action yields/costs.
7. Tune Dominion target or campaign length only if systemic.

The aim is not to make the game trivial.

The aim is to make Standard reliably beatable through sane play.

# 23. Tags and Stats Clarity

If scope allows, add first-pass tag/stat help.

## Tag explanation

Tags are role hints.

Player-facing text:

```text
Tags describe what a game piece tends to interact with. They do not always change stats directly, but they tell you where the piece is useful.
```

Examples:

```text
HEAT CONTROL:
Helps reduce Heat, avoid scrutiny, or manage exposure.

INTEL:
Helps generate, spend, or exploit Intel and Secrets.

INDUSTRIAL:
Associated with Chrome Narrows, money, violence, and high-Heat opportunities.
```

Every non-obvious stat should have a tooltip or help icon.

Examples:

```text
Leverage:
Coercive power over a contact. Higher Leverage can unlock pressure-based options, but often increases social danger.

Volatility:
Instability in a contact relationship. High Volatility makes contact complications more likely.

Obligation:
How much you owe a faction. High Obligation makes faction demands more likely.
```

This is secondary to HandlerBot, but highly aligned with v0.9.

# 34. Summary

v0.9 should solve the most important player-experience problem:

```text
How do I play this game well enough to enjoy it?
```

v0.1 asked:

```text
Can the pressure loop work?
```

v0.2 asked:

```text
Where do I dare do this?
```

v0.3 asked:

```text
Who do I trust to do it?
```

v0.4 asked:

```text
What have I learned, what do I owe, and what can I spend?
```

v0.5 asked:

```text
Who can I use, who can use me back, and what does this connection become?
```

v0.6 asked:

```text
What do I own, what does it produce, and who notices?
```

v0.7 asked:

```text
Which institutions will help me rise, and what will they make me become?
```

v0.8 asked:

```text
What kind of city woke up this time?
```

v0.9 asks:

```text
Can the game teach me how to survive it?
```

That must come before the big immersion pass.

A beautiful unfair game is still unfair.

A complex game that teaches itself can become beloved.
