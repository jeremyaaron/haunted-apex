## Core thesis

> **Power in Haunted Apex should not only come from what you do, where you do it, who you send, or what you know. It should also come from who you are entangled with.**

v0.1 proved the pressure loop.

v0.2 made districts, venues, and rivals matter.

v0.3 made operatives matter.

v0.4 made secrets, debts, favors, and consequences persist through the Black Ledger.

v0.5 should make **contacts** matter.

Not full romance.

Not full diplomacy.

Not a sprawling NPC life simulator.

The goal is narrower:

> **Introduce non-roster characters who can help, betray, stabilize, expose, tempt, or complicate the player based on a few readable relationship pressures.**

The player should begin thinking:

```text
I need to lower Heat. Captain Hollis can help, but I already pressured him twice.

Veyra can get me into the Glass Saint, but every favor from her gives Nyx a longer shadow.

Dr. Mercy can stabilize my crew, but I cannot afford another clean-room bill.

Ciro Moth knows the Ghostline routes, but every time I listen to him, Ruin gets worse.
```

That is the target.

---

# 1. Design Goal

v0.5 introduces the first real **Relationship Web**.

This means adding a new entity class:

```text
Contacts
```

Contacts are not operatives.

Contacts are not rivals.

Contacts are useful outsiders connected to locations, rivals, ledger entries, events, and pressure states.

They provide:

```text
services
event hooks
ledger hooks
pressure relief
new liabilities
run identity
```

The player does not “own” contacts. The player is entangled with them.

---

# 2. What v0.5 Should Prove

v0.5 should answer:

> **Can Haunted Apex make recurring non-crew characters feel strategically useful and emotionally dangerous?**

Target player stories:

```text
“I won because Captain Hollis kept Heat under control, but I had to burn his favor at the worst possible time.”

“I trusted Veyra too much and Nyx used that channel to poison my loyalty.”

“Dr. Mercy saved Juno from breaking, but the debt showed up in the Black Ledger two weeks later.”

“Ciro Moth gave me the route that won the run, and I still regret taking it.”

“I pressured Mina Glass for Intel and permanently soured the best contact I had.”
```

The player should remember contacts as **levers with feelings**, not just buttons.

# 5. Content Philosophy

Contacts are not lore decorations.

A contact must answer:

```text
What do they give me?
What do they cost me?
What pressure do they relieve?
What pressure do they create?
What happens if I overuse them?
Who else are they connected to?
```

A good contact is not purely good or bad.

Bad contact design:

```text
Mina gives +8 Intel.
```

Good contact design:

```text
Mina Glass can open elite nightlife channels for Intel and Dominion, but her proximity to Nyx increases Volatility and can turn favors into social debt.
```

The Contact system should make the city feel socially alive without requiring lots of prose.

```text

```

## Rivals

```text
Competing powers.
Control territory.

```

## Ledger Entries

```text
Secrets, debts, and favors.
Persistent consequences and leverage artifacts.
```

## Contacts

```text
Useful/dangerous outsiders.
Can be cultivated, pressured, or asked for services.
Connect to Ledger entries, rivals, locations, and events.
```

Contacts are the bridge between the Black Ledger and the living city.

# 7. Core Relationship Metrics

Each active contact has four visible metrics.

```ts
type ContactState = {
  id: ContactId;

  trust: number; // 0–100
  leverage: number; // 0–100
  volatility: number; // 0–100
  exposure: number; // 0–100

  status: ContactStatus;

  active: boolean;
  burned: boolean;

  recentInteractions: ContactInteraction[];
  flags: Record<string, boolean | number | string>;
};
```

## Trust

How willing the contact is to help voluntarily.

High Trust:

```text
better services
lower cost
positive events
less chance of backlash
```

Low Trust:

```text
more expensive services
greater chance of refusal/complication
betrayal/pressure events become more likely
```

## Leverage

How much usable pressure the player has over the contact.

High Leverage:

```text
stronger coercive options
more immediate value
can substitute for Trust
```

But high Leverage usually increases:

```text
Volatility
Ruin
future backlash
```

## Volatility

How unstable the relationship is.

High Volatility:

```text
contact-specific events become more likely
services may have extra cost
debts/favors may become messier
```

Volatility is not always bad. It creates opportunities and drama.

## Exposure

How dangerous the connection is to maintain.

High Exposure:

```text
Heat effects are stronger
rivals notice more
contact events may involve surveillance, leaks, or public risk
```

Exposure is especially relevant for officials, liaisons, and rival-linked contacts.

# 8. Derived Contact Status

Contact status is a readable label derived from metrics.

```ts
type ContactStatus =
  | 'cold'
  | 'useful'
  | 'trusted'
  | 'pressured'
  | 'entangled'
  | 'volatile'
  | 'burned';
```

Recommended derivation:

```ts
function deriveContactStatus(contact: ContactState): ContactStatus {
  if (contact.burned) return 'burned';

  if (contact.volatility >= 75) return 'volatile';

  if (contact.trust >= 70 && contact.leverage < 50) return 'trusted';

  if (contact.leverage >= 65 && contact.trust < 50) return 'pressured';

  if (contact.trust >= 50 && contact.leverage >= 50) return 'entangled';

  if (contact.trust < 25 && contact.leverage < 25) return 'cold';

  return 'useful';
}
```

This gives players a quick read without hiding the underlying numbers.

# 11. New Action: Manage Contact

Add one new action.

```text
Manage Contact
```

This action targets a contact and a contact approach/service option.

It costs one command point.

```ts
{
  id: 'manage_contact',
  label: 'Manage Contact',
  commandCost: 1,
  resourceCost: 0,
  effects: {},
  baseRisk: 10,
  requiresTarget: true,
  allowedTargetTypes: ['contact']
}
```

## Contact target model

```ts
type ActionTarget =
  | { type: 'district'; id: DistrictId }
  | { type: 'venue'; id: VenueId }
  | { type: 'rival'; id: RivalId }
  | { type: 'recruit'; id: OperativeId }
  | { type: 'ledger'; entryId: LedgerEntryId; useOptionId: LedgerUseOptionId }
  | { type: 'contact'; contactId: ContactId; optionId: ContactOptionId };
```

The option determines what the action does.

# 12. Contact Options

Every contact should have generated options based on their current state.

Core option types:

```ts
type ContactOptionKind = 'cultivate' | 'pressure' | 'request_service';
```

## Cultivate

Build Trust and reduce Volatility.

General use:

```text
Spend resources/time to make the relationship safer and more cooperative.
```

Default effects:

```ts
{
  trust: +10,
  volatility: -6,
  exposure: +2
}
```

Possible cost:

```text
Resources -600
```

Cultivate should be safe but slow. It should not directly win the game.

## Pressure

Increase Leverage and generate immediate value, but damage the relationship.

General use:

```text
Turn the contact into an asset through coercion, blackmail, or fear.
```

Default effects:

```ts
{
  leverage: +10,
  trust: -6,
  volatility: +8,
  ruin: +2,
  intel: +4
}
```

Pressure is the dangerous shortcut.

## Request Service

Use the contact’s specific function.

Examples:

```text
Clean Passage
Private Room Access
Patch the Crew
Ghostline Route
Absolution Protocol
```

Services should usually cost one or more of:

```text
Trust
Leverage
Resources
Intel
Ledger entry
Rival pressure
Future Debt
```

# 13. Contact Services

A contact service is a reusable option.

```ts
type ContactServiceDefinition = {
  id: ContactServiceId;
  label: string;
  description: string;

  requirements?: ContactServiceRequirement[];

  cost?: {
    resources?: number;
    intel?: number;
    trust?: number;
    leverage?: number;
  };

  effects: PressureDelta;

  contactEffects?: {
    trust?: number;
    leverage?: number;
    volatility?: number;
    exposure?: number;
  };

  ledgerEffects?: LedgerEffect[];

  rivalPressureEffects?: Partial<Record<RivalId, number>>;

  riskModifier?: number;
};
```

## Service affordability

If the player cannot pay the resource/intel cost, the option is disabled.

If the service requires Trust or Leverage and the contact does not have enough, it is disabled.

Example:

```text
Clean Passage
Requires Trust >= 35 or Leverage >= 35
Cost: Resources -700 or Trust -8
Effect: Heat -10
```

For v0.5, keep service choices simple. Do not add multiple payment paths unless implementation is already easy from v0.4 Ledger use options.

# 15. Contact-Linked Ledger Integration

v0.5 should connect some v0.4 Ledger entries to contacts.

Ledger entries gain optional field:

```ts
type LedgerEntry = {
  // existing fields...
  relatedContactId?: ContactId;
};
```

## Examples

```text
Debt: Owes the Liaison
relatedContactId: Veyra Lux, if active

Favor: Checkpoint Captain
relatedContactId: Captain Rafe Hollis, if active

Secret: Dead Channel Trace
relatedContactId: Ciro Moth or Father Static, if active

Debt: Saint’s Payment Trail
relatedContactId: none, unless future contact exists
```

## Rule

Not every Ledger entry needs a contact.

But when a Ledger entry does have a contact, using/resolving it should affect contact state.

Examples:

```text
Settle Debt: Owes the Liaison
Veyra Trust +4, Volatility -6

Ignore Debt Comes Due
Veyra Trust -8, Volatility +10

Call Favor: Checkpoint Captain
Hollis Trust -6, Exposure +5
```

This is how v0.4 and v0.5 interlock.

# 18. Contact Event Scheduling

Contact events use the existing single weekly event slot.

They enter the normal weighted event pool.

Do not add extra event spam.

## Weighting

Contact events become more likely when:

```text
contact Volatility is high
contact Trust is low
contact Exposure is high
contact was used recently
related rival pressure is high
related district/venue was targeted recently
related Ledger entry exists
```

## Once-per-run

Most contact signature events should happen at most once per run.

Generic contact events can repeat lightly, but should be downweighted after firing.

## OperatorBot

Use contacts when they solve a specific problem.

# 22. Balance Goals

Preserve broad v0.4 tuning.

Expected:

```text
OperatorBot: 55–75% win rate
CautiousBot: low win rate, usually Dominion shortfall
RandomBot: 5–12% win rate
AggressiveBot: viable but volatile
GreedyBot: viable but swingy
```

Contact-specific targets:

```text
Average Manage Contact uses per run: 1–3
Average contact-specific events per run: 0–2
At least one contact metric changes meaningfully in most runs
Contacts should help stabilize bad runs but create future hooks
No single contact should be mandatory for OperatorBot
Pressure should be powerful but visibly corrosive
Cultivate should be useful in longer or strained runs, but not always necessary
```

The player should be able to win while ignoring contacts, but contacts should make runs more interesting and give comeback paths.

# 24. Technical Guidance

## Avoid hardcoding contacts into event logic

Bad:

```ts
if (contact.id === 'contact_veyra_lux') {
  state.pressures.intel += 8;
}
```

Better:

```ts
resolveContactService(contactServiceDefinition, context);
applyContactEffects(contactId, service.contactEffects);
applyLedgerEffects(service.ledgerEffects);
```

Some hand-authored event definitions are fine, but the engine should be generic.

## Suggested modules

```text
content/contacts.ts
content/contactServices.ts
content/contactEvents.ts

engine/contacts/generateContacts.ts
engine/contacts/contactSelectors.ts
engine/contacts/deriveContactStatus.ts
engine/contacts/resolveContactOption.ts
engine/contacts/applyContactEffects.ts
engine/contacts/contactEventEligibility.ts
engine/contacts/contactReports.ts
```

## GameState addition

```ts
type GameState = {
  // existing fields...
  contacts: Record<ContactId, ContactState>;
  activeContactIds: ContactId[];
};
```

## Recent contact activity

```ts
type ContactInteraction = {
  week: number;
  optionId: ContactOptionId;
  kind: ContactOptionKind;
  effectsSummary: {
    trust?: number;
    leverage?: number;
    volatility?: number;
    exposure?: number;
  };
};
```

Keep this lightweight.
