## Foundation Vision Document

## Core thesis

> **Each run should stop feeling like the same board with different pieces and start feeling like a specific crisis in a specific city.**

v0.1 proved the pressure loop.

v0.2 made territory and rivals matter.

v0.3 made operatives matter.

v0.4 made secrets, debts, favors, and consequences persist.

v0.5 made contacts useful, dangerous, and socially sticky.

v0.6 made ownership matter through fronts.

v0.7 made institutions matter through factions and accords.

v0.8 should make the **whole run** matter.

The player should no longer think only:

```text
I started another Haunted Apex run.
```

They should think:

```text
This is a Ghostline Signal run.

Juno is in the starting roster.
Father Static is active.
Ghostline Communion is active.
Secrets are everywhere.
Ruin is already rising.
The Surveillance Den is on the table.
This run wants me to get powerful by listening to things that should stay dead.
```

The goal is not a sprawling procedural universe yet.

The goal is narrower:

> **Introduce campaign tensions that assemble existing systems into coherent run identities.**

---

# 1. Design Goal

v0.8 introduces **Campaign Tensions**.

A Campaign Tension is a run-level scenario profile that influences:

```text
starting pressures
active rivals
active factions
active contacts
front opportunities
roster generation weights
event weights
Ledger discovery flavor
run summary copy
```

It does not replace the current game.

It gives the current game a stronger opening premise.

Current game:

```text
Seeded run with operatives, contacts, fronts, factions, rivals, events.
```

v0.8 game:

```text
Seeded run organized around a city crisis.
```

---

# 2. What v0.8 Should Prove

v0.8 should answer:

> **Can Haunted Apex generate runs that feel meaningfully different before the player takes the first action?**

Target player stories:

```text
“My Corp Crackdown run forced me to care about Ashline from Week 1.”

“The Ghostline Signal run made Juno incredible and terrifying.”

“Nightlife War with Veyra, Nyx, and Velvet House active felt like playing inside a knife fight in a private room.”

“Industrial Cut gave me money fast, but Heat went insane.”

“Dirty Capital handed me resources early and then buried me in obligations.”
```

The player should start developing preferences:

```text
I love Ghostline Signal runs.
I hate Corp Crackdown unless I start with Hollis.
Industrial Cut is greedy-bot heaven.
Nightlife War is the most Haunted Apex thing in the game.
```

That is the right kind of replayability.

# 5. Content Philosophy

A campaign tension is not just a modifier.

It is a **run premise**.

Bad campaign design:

```text
Scenario: Hard Mode
Heat +10
```

Good campaign design:

```text
Corp Crackdown

Ashline Bureau has turned the lower city into a pricing engine for visibility. Heat starts high, Ashline Suspicion starts high, Front Inspection and Corp Patrol events become more likely, but Ashline accords and security contacts become more valuable.
```

A good campaign tension should answer:

```text
What is wrong with the city this time?
Which systems become more important?
Which characters/factions are favored?
Which strategy is tempting?
Which pressure becomes dangerous?
What kind of story will this run probably produce?
```

# 6. Run Assembly vs Existing Generators

v0.8 should centralize run generation.

Right now, the game likely has separate generation paths for:

```text
operatives
hire pool
contacts
front opportunities
factions
rivals
maybe events / state initialization
```

v0.8 should introduce a single high-level entry point:

```ts
function assembleNewRun(config: NewRunConfig): GameState;
```

This function owns the sequence:

```text
1. Create RNG from seed.
2. Select Campaign Tension.
3. Generate city identity.
4. Initialize base pressures.
5. Apply campaign starting modifiers.
6. Select active factions.
7. Select active rivals.
8. Select active contacts.
9. Select starting roster and hire pool.
10. Select front opportunities.
11. Create starting owned front.
12. Initialize Ledger/Accords/Fronts/Contacts/Factions/Rivals.
13. Create opening briefing.
14. Return GameState.
```

The campaign tension influences the downstream generators.

This is the technical heart of v0.8.

# 7. Campaign Tension Model

## Static definition

```ts
type CampaignTensionDefinition = {
  id: CampaignTensionId;
  name: string;
  subtitle: string;

  roleTags: CampaignRoleTag[];

  description: string;
  openingBriefing: string;

  startingPressureDelta?: PressureDelta;

  factionModifiers?: Partial<Record<FactionId, FactionMetricDelta>>;
  rivalPressureModifiers?: Partial<Record<RivalId, number>>;
  contactMetricModifiers?: Partial<Record<ContactId, ContactMetricDelta>>;

  generationBias: CampaignGenerationBias;

  eventWeightModifiers?: CampaignEventWeightModifier[];

  secretDiscoveryModifier?: number;

  runSummaryFlavor?: {
    victoryLine?: string;
    lossLine?: string;
    epitaphTemplates?: string[];
  };
};
```

## Campaign role tags

```ts
type CampaignRoleTag =
  | 'heat'
  | 'nightlife'
  | 'ghostline'
  | 'industrial'
  | 'capital'
  | 'fronts'
  | 'ledger'
  | 'contacts'
  | 'factions'
  | 'rivals'
  | 'ruin'
  | 'resources'
  | 'dominion';
```

## Generation bias

```ts
type CampaignGenerationBias = {
  requiredFactionIds?: FactionId[];
  weightedFactionIds?: Partial<Record<FactionId, number>>;

  requiredRivalIds?: RivalId[];
  weightedRivalIds?: Partial<Record<RivalId, number>>;

  requiredContactIds?: ContactId[];
  weightedContactIds?: Partial<Record<ContactId, number>>;

  weightedOperativeIds?: Partial<Record<OperativeId, number>>;
  weightedOperativeTags?: Partial<Record<OperativeRoleTag, number>>;

  weightedFrontDefinitionIds?: Partial<Record<FrontDefinitionId, number>>;
  weightedFrontTags?: Partial<Record<FrontRoleTag, number>>;

  weightedEventTags?: Partial<Record<EventTag, number>>;
};
```

## Event weight modifiers

```ts
type CampaignEventWeightModifier = {
  eventId?: EventId;
  eventTag?: string;
  weightDelta: number;
};
```

Use event tags wherever possible. Avoid hardcoding every event.

# 12. Campaign Modifiers

Keep campaign modifiers simple.

v0.8 should use three categories:

```text
Starting modifiers
Generation bias
Event weight modifiers
```

Only one campaign should have a simple runtime mechanic:

```text
Ghostline Signal: +8 percentage points to targeted Gather Intel secret discovery.
```

Do not add many campaign-specific rules yet.

The point is assembly, not bespoke scenarios.

# 13. Campaign-Aware Generation

## Factions

Current rule likely:

```text
Ashline always active.
Seeded additional factions.
```

v0.8 should allow campaign required factions.

Generation rule:

```text
1. Start with always-active factions.
2. Add required campaign factions.
3. Fill remaining active slots by weighted seed selection.
4. Maintain active faction count if possible.
```

If active faction count is 4 and required factions exceed slots, allow campaign to override the count only if necessary. For v0.8, avoid tensions that require too many.

## Rivals

Generation rule:

```text
1. Add required campaign rivals.
2. Fill remaining rival slots by weighted selection.
3. Avoid duplicates.
```

If the current game always has fixed rivals, v0.8 should at least weight which rival pressure starts higher and which rival is featured.

## Contacts

Generation rule:

```text
1. Add required campaign contacts if any.
2. Fill active contacts with weighted selection while preserving role coverage.
3. Avoid invalid contact sets.
```

## Operatives

Do not force operatives unless needed.

Prefer weighting.

Example:

```text
Ghostline Signal makes Juno more likely, not guaranteed.
```

Reason:

```text
Starting roster variety is still important.
```

## Front Opportunities

Generation rule:

```text
1. Preserve The Pale Circuit starting front.
2. Weight front opportunities by campaign.
3. Preserve opportunity count and front cap rules.
```

# 14. Event Weighting

Campaign event weights should be additive.

Existing event selection can consume a context object.

Add:

```ts
type EventWeightContext = {
  // existing...
  campaignTensionId?: CampaignTensionId;
  campaignEventWeightModifiers?: CampaignEventWeightModifier[];
};
```

Event weight calculation:

```ts
weight = baseWeight
  + state/context modifiers
  + rival/contact/front/faction modifiers
  + campaign modifiers
```

Clamp minimum weight:

```text
Do not allow campaign modifiers to make eligible events impossible unless explicitly disabled.
```

Suggested:

```ts
weight = Math.max(0, weight);
```

# 18. Agent Updates

Agents should become campaign-aware.

## OperatorBot

Adjust strategy based on tension.

### Corp Crackdown

```text
Prioritize Heat control.
Value Ashline accords more.
Avoid high-exposure fronts unless stabilized.
Use Ledger/Favors for Heat relief earlier.
```

### Nightlife War

```text
Value social contacts, Intel, and Loyalty protection.
Avoid overfeeding Nyx pressure.
Use Velvet/liaison opportunities when Heat is manageable.
```

### Ghostline Signal

```text
Use targeted Gather Intel more often.
Watch Ruin.
Value Echo Saint / Father Static / Ruin relief.
Use secrets more aggressively if behind.
```

### Industrial Cut

```text
Accept more money/Dominion risk early.
Watch Heat and Knox pressure.
Use bribes/Lay Low/front cooling earlier.
```

### Dirty Capital

```text
Invest in fronts earlier.
Use capital to accelerate Dominion.
Watch faction Obligation and Debt Comes Due.
```

## CautiousBot

Campaign-aware but still conservative.

Expected:

```text
Still often misses Dominion.
May perform slightly better in Corp Crackdown and Dirty Capital.
```

## AggressiveBot

Should exploit:

```text
Industrial Cut
Nightlife War
Chrome/Velvet acceleration
```

## GreedyBot

Should exploit:

```text
Dirty Capital
Industrial Cut
fronts and money accords
```

## RandomBot

Still diagnostic only.

# 20. Balance Goals

Preserve broad v0.7 tuning, but evaluate by campaign.

Expected global profile:

```text
OperatorBot: 55–75% win rate overall
CautiousBot: usually fails by Dominion shortfall
RandomBot: diagnostic, 0–10% acceptable
AggressiveBot: viable but volatile
GreedyBot: viable but swingy
```

Campaign-specific OperatorBot target:

```text
Each Campaign Tension should give OperatorBot at least 40% win rate.
No Campaign Tension should give OperatorBot above 85% win rate.
```

Campaign-specific style targets:

```text
Corp Crackdown:
Higher Heat losses than average.

Nightlife War:
Higher Loyalty/contact/rival complications than average.

Ghostline Signal:
Higher Ruin and Ledger activity than average.

Industrial Cut:
Higher Resources, higher Heat, higher aggressive wins/losses.

Dirty Capital:
Higher front usage, higher Debt/Obligation events.
```

The goal is not equal difficulty. It is distinct texture.

# 21. Regret Patterns in v0.8

## Good regret

```text
I treated Corp Crackdown like a normal run and drowned in Heat.

I leaned into Ghostline secrets and forgot Ruin was not just flavor.

Dirty Capital gave me money so early that I overbuilt fronts and spent the rest of the run paying for it.

Nightlife War made every social option look delicious and every one came with teeth.

Industrial Cut made me rich enough to win and loud enough to almost lose.
```

## Bad regret

```text
The campaign tension was invisible after the briefing.

A scenario forced a bad roster with no counterplay.

Required content made every run too samey.

Campaign modifiers created unexplained pressure changes.

The player could not tell why events were repeating.

A campaign tension felt like a difficulty preset instead of a story premise.
```

The tension should shape the run without hijacking it.

# 22. Technical Guidance

## Centralize assembly

v0.8 is partly a refactor milestone.

The game should have one obvious new-run path.

Suggested modules:

```text
content/campaignTensions.ts
content/cityNames.ts

engine/campaign/assembleNewRun.ts
engine/campaign/selectCampaignTension.ts
engine/campaign/generateCityIdentity.ts
engine/campaign/applyCampaignStartingModifiers.ts
engine/campaign/applyCampaignGenerationBias.ts
engine/campaign/campaignSelectors.ts
engine/campaign/campaignReports.ts
```

## Avoid campaign-specific hardcoding in unrelated systems

Bad:

```ts
if (state.campaign.tensionId === 'ghostline_signal') {
  state.pressures.ruin += 2;
}
```

Better:

```ts
applyCampaignModifiers(state, context);
```

Some special-case behavior is acceptable for one or two scenario-specific hooks, but the main system should be data-driven.

## Versioned save invalidation

v0.8 should invalidate v0.7 saves.

Reason:

```text
Run assembly and GameState shape change.
```

No migration required.
