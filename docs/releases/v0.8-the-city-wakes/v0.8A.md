# Haunted Apex v0.8.0 — The City Wakes Clarifications

## 1. Should players be able to choose a Campaign Tension manually?

Yes.

New Run should offer both:

```text
Random Campaign Tension
Specific Campaign Tension
```

Default should be:

```text
Random
```

Player-facing behavior:

```text
Campaign Tension: Random
Seed: optional
```

If the player wants a specific scenario, they can select one.

This is useful for:

```text
playtesting
agent harness runs
players replaying favorite tension types
debugging
future challenge seeds
```

Locked rule:

```text
v0.8 New Run supports Random by default and manual Campaign Tension selection.
```

---

## 2. Should Campaign Tension be selected from seed alone, or seed plus config?

Use:

```ts
newGame({
  seed?: string;
  campaignTensionId?: CampaignTensionId;
})
```

Rules:

```text
If campaignTensionId is provided:
  use that tension.

If campaignTensionId is absent:
  select Campaign Tension deterministically from seed.

If seed is absent:
  generate seed, then select tension deterministically from generated seed.
```

This means:

```text
Same seed + no explicit tension = same tension.
Same seed + explicit tension = same tension override.
```

Locked rule:

```text
Campaign selection is deterministic and config-driven.
```

---

## 3. Are active rivals still always both active?

For v0.8: yes.

Do **not** introduce active/inactive rivals yet.

The current rival pool is too small for active/inactive rival generation to add much value. It would add state complexity without enough payoff.

So for v0.8:

```text
Nyx Ardent and Knox Marrow remain active.
Campaign tensions modify rival pressure, event weighting, and briefing emphasis.
```

Campaign fields like `requiredRivalIds` and `weightedRivalIds` can exist in the definition model for future compatibility, but in v0.8 they should function as:

```text
featured rival indicators
starting pressure modifiers
event-weight modifiers
report grouping/context
```

Locked rule:

```text
v0.8 does not introduce active/inactive rivals. Campaigns feature or pressure rivals, but both current rivals remain present.
```

Future version:

```text
Once there are 4–6 rivals, active rival selection becomes valuable.
```

## 4. Should featured content show guaranteed content only, or also weighted/likely content?

Separate them.

Use two categories:

```text
Guaranteed / Active
Favored / More Likely
```

Example:

```text
Guaranteed:
- Ghostline Communion active
- Ghostline Signal starting modifiers applied

Favored:
- Juno Hex more likely in roster
- Ciro Moth / Father Static more likely as contacts
- Surveillance Den more likely as front opportunity
- Ledger / weird events weighted upward
```

Do not call weighted content “featured” unless it is actually guaranteed or already active in the generated run.

Better briefing labels:

```text
Active This Run
Favored By This Tension
Pressure Pattern
```

Locked rule:

```text
The briefing distinguishes guaranteed/active content from favored/weighted content.
```

## 5. How visible should exact campaign modifiers be?

Show exact numbers for applied starting modifiers.

The game is already mechanical and strategic. Players should see what changed.

Good:

```text
Starting Effects:
Heat +12
Intel -2
Ashline Suspicion +12
```

For generation bias, do **not** show raw weights.

Use qualitative labels:

```text
Favored:
Heat-control operatives
Security contacts
Heat-control fronts
Corp Patrol / Front Inspection events
```

So:

```text
Exact numbers for applied state changes.
Qualitative language for probability/weighting.
```

Locked rule:

```text
Starting pressure/faction/rival modifiers are shown exactly. Generation and event biases are shown qualitatively.
```

---

## 6. Should opening briefing appear after loading an existing save?

Use `openingBriefingShown`.

Rules:

```text
New run:
openingBriefingShown = false
show briefing

Player dismisses:
openingBriefingShown = true

Load save:
if openingBriefingShown === false, show briefing
if openingBriefingShown === true, do not auto-show
```

But the player should be able to reopen it.

Minimum UI:

```text
Header shows: City · Campaign Tension
Button/link: Campaign
```

The Campaign button opens the briefing/details panel again.

Locked rule:

```text
Briefing auto-shows only until dismissed, but campaign details remain accessible during the run.
```

## 7. Should campaign modifiers be represented in event log?

Yes.

Add a concise opening log entry.

Example:

```text
Campaign Applied: Corp Crackdown
Heat +12, Intel -2, Ashline Suspicion +12.
```

For more complex campaigns:

```text
Campaign Applied: Nightlife War
Dominion +3, Loyalty -4, Heat +4, Nyx Pressure +15.
```

This helps:

```text
debugging
player clarity
run history
agent report traceability
```

Do not spam the log with every generation bias. Keep it to actual applied deltas.

Locked rule:

```text
Campaign starting modifiers generate one concise opening log entry.
```

## 8. Does Ghostline Signal’s `+8` secret discovery bonus apply only to targeted Gather Intel?

Yes. Strictly.

Rule:

```text
Ghostline Signal bonus applies only to targeted Gather Intel secret discovery chance.
```

Not:

```text
untargeted Gather Intel
event-created secrets
Ledger event rewards
contact service secrets
accord-created secrets
front event secrets
```

Implementation:

```ts
if (
  state.campaign.tensionId === 'campaign_ghostline_signal' &&
  order.actionId === 'gather_intel' &&
  order.target
) {
  secretDiscoveryChance += 8;
}
```

Then clamp as usual.

Preview should show it.

Example:

```text
Secret Chance: 34%
Campaign Bonus: Ghostline Signal +8%
```

Locked rule:

```text
Ghostline Signal’s secret discovery bonus applies only to targeted Gather Intel and appears in preview.
```

## 9. Should campaign-specific report grouping be only harness-level, or also player run summary?

Both, but at different depth.

## Player run summary

Concise.

Include:

```text
City
Campaign Tension
Seed
Result
Final pressures
1–3 campaign-relevant notes
```

Example:

```text
Campaign Notes:
Ghostline Signal
Secrets discovered: 4
Ruin gained: 17
Ledger entries used: 3
```

## Harness reports

Detailed.

Group by:

```text
campaign tension
agent strategy
win rate
loss cause
final pressures
action usage
event frequency
system usage
```

Locked rule:

```text
Player summary is concise; harness reports provide full campaign grouping and analysis.
```

## 10. Are the five tensions locked for v0.8?

Yes. Lock these five:

```text
Corp Crackdown
Nightlife War
Ghostline Signal
Industrial Cut
Dirty Capital
```

They are distinct enough and map well to existing systems.

## ID references

The TDD should use the actual existing repo IDs.

The vision document names are conceptual labels. If an ID differs in code, use the code’s actual ID.

Example:

```text
Vision: Mina Glass
Code: contact_mina_glass or whatever current registry uses
```

Use current registry ID.

If referenced content does not exist:

```text
Do not create major new content just to satisfy a campaign bias.
```

Instead:

1. Map to the closest existing entity if appropriate.
2. Omit the bias if no close equivalent exists.
3. Add missing content only if it is tiny and already aligned with prior versions.

For v0.8, the goal is not expanding contacts/operatives/fronts. It is assembling them.

Locked rule:

```text
The five Campaign Tensions are locked. TDD must normalize references to existing content IDs and avoid unnecessary content expansion.
```
