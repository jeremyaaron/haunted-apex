# Haunted Apex v0.3.0 — The Roster

## Foundation Vision Document

## Version

```text
v0.3.0 — The Roster
```

## Previous milestone

```text
v0.2.0 — Rival Territory
```

## Core thesis

> **The same action should feel different not only because of where it happens, but because of who you send and what that choice does to them over time.**

v0.1 proved the pressure loop.

v0.2 made locations and rivals matter.

v0.3 should make **operatives** matter as people, not just stat modifiers.

The goal is not yet a full relationship system, faction diplomacy system, romance system, or procedural narrative engine. The goal is narrower and stronger:

> **Players should begin forming preferences, grudges, attachments, and stories around individual operatives.**

After v0.3, a player should say things like:

```text
“Mara saved that run.”

“Juno is powerful, but she always drags me into Ruin.”

“I hate getting Knox early unless I’m playing aggressive.”

“Saint is useful, but I never fully trust him.”

“Iris changes the whole early game if Nyx is active.”
```

That is the beginning of fan obsession.

---

# 1. Design Goal

v0.3 introduces a curated operative roster and makes crew composition a strategic variable.

The player no longer always starts with the same three operatives. Instead, each campaign draws from a curated pool using seeded RNG, while preserving role coverage so the opening state remains playable.

The roster becomes a set of learnable game pieces.

## v0.2 structure

```text
Action + Operative + Target → Outcome + Rival Reaction
```

## v0.3 structure

```text
Action + Operative Identity + Operative Condition + Target → Outcome + Consequence
```

The important expansion is **identity over time**.

An operative should not just answer:

```text
What stat do they modify?
```

They should answer:

```text
What kind of strategy do they invite?
What pressure do they help manage?
What pressure do they create?
What happens if I overuse them?
What are they good at under stress?
What makes them dangerous?
```

---

# 2. What v0.3 Should Prove

v0.3 should answer:

> **Can Haunted Apex generate player stories from a small curated roster?**

Not prose-heavy story. Gameplay story.

Examples:

```text
I kept using Juno in Ghostline Market until her stress broke 80, then a ghost-tech event spiked Ruin and gave me the Intel I needed to win.

I recruited Knox because I was behind on Dominion, but his loud bonuses kept pushing Heat into lockdown.

I got Mara and Orchid together, so I played a low-Heat Intel/control strategy and avoided Knox entirely.

Saint’s discount bribes saved me twice, then his concealed debt event hit when I was nearly broke.
```

That is the target.

# 5. Content Philosophy

v0.3 should adopt this content model:

> **Curated roster, seeded assembly.**

Do not make operatives fully random.

Do not make every campaign use the same fixed crew.

Instead:

```text
Operatives are authored.
Campaigns are assembled.
Runs are seeded.
Outcomes emerge.
```

Each operative is a recognizable game piece.

The player learns them across runs.

The campaign generator decides which pieces appear.

# 6. Operative Model

## Static Definition

Operative definitions should live in a static content registry.

```ts
type OperativeDefinition = {
  id: OperativeId;
  name: string;
  archetype: OperativeArchetype;

  rarity: OperativeRarity;
  roleTags: OperativeRoleTag[];

  baseStats: OperativeStats;

  signatureTraitId: TraitId;
  liabilityTraitId?: TraitId;

  affinities: OperativeAffinity[];
  stressProfile: StressProfile;

  eventIds: EventId[];

  flavor: {
    dossier: string;
    quote?: string;
    visualTags?: string[];
  };
};
```

## Mutable State

Campaign-specific operative state lives inside `GameState`.

```ts
type OperativeState = {
  id: OperativeId;

  loyalty: number;
  stress: number;

  status: OperativeStatus;

  revealedTraits: TraitId[];
  hiddenFlags: Record<string, boolean | number | string>;

  weeksAssigned: number;
  recentAssignments: RecentAssignment[];
};
```

## Stats

Keep the current stat model unless there is a strong implementation reason to change it.

```ts
type OperativeStats = {
  violence: number;
  charm: number;
  tech: number;
  subtlety: number;
};
```

Do not add too many stats yet.

Potential future stats like discipline, nerve, occult, empathy, or logistics can wait.

# 7. Rarity

Rarity is not monetization. It is campaign texture.

```ts
type OperativeRarity = 'common' | 'uncommon' | 'rare' | 'legendary';
```

## Meaning

```text
Common:
Reliable, straightforward, appears often.

Uncommon:
More specialized, stronger identity.

Rare:
Powerful or strange, usually with a meaningful liability.

Legendary:
Run-warping, should appear rarely and may not be included in v0.3 unless trivial.
```

## v0.3 recommendation

Use only:

```text
common
uncommon
rare
```

Save legendary for later unless one obvious candidate emerges.

# 8. Role Tags

Role tags are used for roster generation, bot strategy, UI filtering, and reports.

```ts
type OperativeRoleTag =
  | 'intel'
  | 'social'
  | 'violence'
  | 'tech'
  | 'heat_control'
  | 'money'
  | 'ruin'
  | 'stability'
  | 'rival_pressure'
  | 'recruitment';
```

Each operative should have 2–4 role tags.

Example:

```ts
{
  id: 'op_mara_voss',
  roleTags: ['intel', 'heat_control', 'stability']
}
```

The generator should use tags to avoid broken starting rosters.

# 9. Starting Roster Generation

Each new campaign should start with three operatives.

```ts
type RosterGenerationConfig = {
  startingRosterSize: 3;
  hirePoolSize: 4;
  allowRareStartingOperatives: boolean;
};
```

## Required coverage

Every starting roster should include:

```text
At least one intel/subtlety-capable operative.
At least one social/charm-capable operative.
At least one money/violence/utility operative.
```

In code, use tags rather than hardcoded stat thresholds if possible.

```ts
const REQUIRED_STARTING_TAG_GROUPS = [
  ['intel', 'tech'],
  ['social', 'heat_control'],
  ['violence', 'money', 'stability'],
];
```

A generated roster is valid if every required group is represented at least once.

## Avoid

```text
Three pure violence operatives.
Three fragile high-Ruin operatives.
No one usable for Heat management.
No one usable for Intel.
No one usable for expansion.
```

The player should experience variety, not accidental unwinnability.

# 10. Hire Pool Generation

The hire pool should be seeded and visible from the start.

Size:

```text
4 candidates
```

Recruit action should select from this pool.

Possible v0.3 behavior:

```text
Recruit Operative opens hire pool.
Player chooses one candidate.
Candidate joins roster.
Candidate removed from hire pool.
No replacement candidate in v0.3 unless trivial.
```

This is better than random recruit because it adds strategic planning.

The player can think:

```text
I did not start with a strong Heat-control operative, but Saint is in the hire pool.
I can spend resources early to stabilize.
```

## Recruit action update

Current v0/v0.2 recruit action can become:

```text
Recruit Operative
Cost: 1 command point, 1600 resources, -4 loyalty, +3 dominion
Select one candidate from hire pool.
```

If implementing a modal is too much, use a selected recruit target.

```ts
type ActionTarget =
  | { type: 'district'; id: string }
  | { type: 'venue'; id: string }
  | { type: 'rival'; id: string }
  | { type: 'recruit'; id: OperativeId };
```

For v0.3, `Recruit Operative` should probably require a recruit target.

# 11. Operative Affinities

Affinities are action/target preferences that make operatives feel distinct.

```ts
type OperativeAffinity = {
  actionId?: ActionId;
  targetTag?: string;
  districtTag?: string;
  rivalId?: RivalId;

  effects?: PressureDelta;
  riskModifier?: number;
  stressModifier?: number;
  rivalPressureModifier?: number;
};
```

Examples:

```ts
const maraAffinities = [
  {
    actionId: 'gather_intel',
    effects: { heat: -1 },
    riskModifier: -3,
  },
  {
    targetTag: 'nightlife',
    effects: { intel: 2 },
    riskModifier: -1,
  },
];
```

```ts
const junoAffinities = [
  {
    targetTag: 'memory',
    effects: { intel: 4, ruin: 2 },
    stressModifier: 4,
  },
  {
    actionId: 'gather_intel',
    effects: { intel: 3, ruin: 1 },
  },
];
```

Affinities should be applied after base action and target modifiers.

Resolution shape:

```text
Base action effects
→ operative trait modifiers
→ operative affinity modifiers
→ target modifiers
→ risk calculation
→ stress changes
→ rival pressure
```

Or, if current v0.2 ordering is already implemented:

```text
Base action
→ operative modifiers
→ target modifiers
→ affinity modifiers
```

The exact order matters less than being consistent and testable.

# 13. Stress Thresholds

Stress should become more legible and consequential.

```ts
type StressTier = 'stable' | 'strained' | 'unstable' | 'breaking';
```

Thresholds:

```ts
function stressTier(stress: number): StressTier {
  if (stress < 40) return 'stable';
  if (stress < 60) return 'strained';
  if (stress < 80) return 'unstable';
  return 'breaking';
}
```

## Mechanical effects

```text
Stable:
No penalty.

Strained:
+2 risk on assigned actions.

Unstable:
+5 risk on assigned actions.
Eligible for stress events.

Breaking:
+10 risk on assigned actions.
Eligible for severe stress events.
May refuse dangerous assignments.
```

## Stress recovery

Keep current recovery:

```text
Idle: -2 stress/week
Lay Low assigned: -8 stress
```

Potential v0.3 addition:

```text
If Loyalty >= 75, idle recovery improves by 1.
If Heat >= 75, idle recovery worsens by 1.
```

Only add this if easy.

# 18. Simulation Agent Updates

The five agents should become roster-aware.

Existing agents:

```text
random
greedy
aggressive
cautious
operator
```

## OperatorBot

Should consider:

```text
operative stress
operative skill fit
operative affinity
role coverage
hire pool gaps
rival/target risk
```

OperatorBot should recruit if:

```text
current roster lacks useful role coverage
or best available candidate fills current strategic problem
or active operatives are too stressed
```

Example:

```text
If Heat is high and Saint/Rook/Vant is in hire pool, recruit.
If Dominion is behind and Knox is in hire pool, recruit if Heat can survive.
If Ruin is high and Echo Saint is in hire pool, recruit.
```

## CautiousBot

Should prefer:

```text
low-stress assignments
low-risk operatives
heat-control operatives
safe targets
Lay Low with breaking operatives
```

Likely still fails Dominion target.

## AggressiveBot

Should prefer:

```text
highest Dominion effects
violence/money operatives
risky targets
ignores stress until severe
```

## GreedyBot

Should prefer:

```text
resources
money tags
Zero Mercy / Black Halo
operatives with money/violence affinity
```

## RandomBot

Random valid action, target, and operative.

No intelligence.

# 20. Balance Goals

v0.3 should preserve the current broad tuning profile.

Expected:

```text
OperatorBot:
55–75% win rate

CautiousBot:
Low win rate, usually Dominion shortfall

RandomBot:
5–12% win rate

AggressiveBot:
Viable but volatile

GreedyBot:
Viable but swingy
```

New roster-specific goals:

```text
No starting roster should make OperatorBot win rate below 40%.
No single operative should be mandatory for sane play.
No single operative should be auto-recruit in every run.
High-power operatives should create identifiable pressure elsewhere.
Stress events should trigger often enough to be seen, not so often they dominate.
```

Initial target:

```text
Operative-specific event appears in 25–45% of runs.
At least one operative reaches Strained in most runs.
Breaking should happen sometimes, but not every sane run.
```

# 21. Regret Patterns in v0.3

v0.3 should create regret through people.

## Good regret examples

```text
I overused Juno because the Intel was too good.
I sent Knox to solve everything and now Heat is impossible.
I ignored stress because the numbers looked fine.
I recruited a specialist too late.
I picked the high-rarity operative instead of the one my run actually needed.
I protected Mara when I should have exploited the debt.
```

## Bad regret examples

```text
An operative randomly betrayed me with no warning.
The best operative was hidden from the UI.
A stress event fired after one assignment.
A roster generated with no way to manage Heat.
```

The player should feel they made understandable tradeoffs.

# 22. Technical Implementation Guidance

## Avoid hardcoding operative identities into engine logic

Bad:

```ts
if (operative.id === 'op_juno_hex') {
  state.pressures.ruin += 2;
}
```

Better:

```ts
applyTraitModifiers(operative, context);
applyAffinityModifiers(operative, context);
checkOperativeEventTriggers(operative, state);
```

The system should support new operatives through content definitions.

# 27. Summary

v0.3.0 should make Haunted Apex more replayable and more emotionally sticky.

The current game already has:

```text
pressure
strategy
targets
rivals
tuning agents
```

Now it needs:

```text
people
```

Not fully simulated people.

Not novel-length characters.

But enough identity that a player starts to care who is in the chair when they choose an action.

The v0.3 question is:

> **Who do you send, and what does sending them cost?**

If v0.2 made the player ask:

```text
Where do I dare do this?
```

v0.3 should make them ask:

```text
Who do I trust to do it?
```
