# Haunted Apex v0.3.0 — Locked Mechanical Decisions

## 1. Final roster

Use all ten proposed operatives.

```text
Mara Voss
Juno Hex
Saint Calder
Iris Vale
Knox Riven
Orchid Seven
Vant Black
Echo Saint
Rook Vale
Mother Neon
```

Mother Neon should be included as **rare**, not legendary. “Legendary” can be saved for later once the game has enough content for truly run-warping pieces.

---

## 2. Rarity rules

Use:

```text
common
uncommon
rare
```

Starting roster rule:

```text
At most one rare operative may appear in the starting roster.
```

Hire pool rule:

```text
Additional rares may appear in the hire pool.
```

This gives rare operatives excitement without letting a run start with a stacked roster.

---

## 3. Starting Loyalty and Stress

Use authored values per operative.

These are identity levers, not generic defaults.

Example:

```ts
type OperativeDefinition = {
  id: OperativeId;
  name: string;
  baseStats: OperativeStats;
  startingLoyalty: number;
  startingStress: number;
  // ...
};
```

This lets you express character immediately.

Examples:

```text
Mara: high loyalty, low-medium stress
Juno: medium loyalty, elevated stress
Saint: medium-high loyalty, low stress, suspicious liability
Knox: medium loyalty, low stress, high volatility through traits
Mother Neon: high loyalty/stability, rare but debt-heavy
```

---

## 4. Roster cap

Keep max roster size at five.

```text
Starting roster: 3
Hire pool: 4
Max roster: 5
```

That means only two hires are possible.

Good. That makes the hire pool a strategic choice instead of a collection checklist.

---

## 5. Trait visibility

Show both:

```text
signature traits
mechanical liability traits
```

immediately.

Keep hidden:

```text
specific event triggers
narrative flags
once-per-run consequences
secret counters
```

Players should be able to reason strategically. The surprise should be in timing and consequence, not in hiding basic mechanical risk.

So:

```text
Visible: “Probably Lying — Low resources may cause Saint’s concealed debt to surface.”
Hidden: exact trigger formula and event weighting.
```

---

## 6. Breaking operatives

Remove automatic unavailability at 80 Stress.

Breaking operatives remain assignable.

At `Breaking`:

```text
+10 action risk
eligible for severe operative events
```

Do **not** add random refusal in v0.3.

That keeps player agency intact. If the player sends a breaking operative anyway, the consequence belongs to the player.

---

## 7. Stress risk calculation

Replace the continuous stress-based risk calculation with tier modifiers.

```ts
function stressRiskModifier(stress: number): number {
  if (stress < 40) return 0; // Stable
  if (stress < 60) return 2; // Strained
  if (stress < 80) return 5; // Unstable
  return 10; // Breaking
}
```

This is much more legible.

Avoid stacking continuous stress risk plus tier risk. That would make stress harder to tune and harder for players to understand.

---

## 8. Operative events

Operative-specific events use the existing single weekly event slot.

They enter the weighted event pool when eligible.

Rule:

```text
One weekly event total.
Eligible operative events compete with city/rival/location events.
Each signature operative event may occur at most once per run.
```

This is the right constraint. It prevents v0.3 from overwhelming the player with event spam.

---

## 9. Event eligibility

Operative events only trigger for current roster members.

Not hire candidates.

General rule:

```text
Only current crew members can trigger operative-specific events.
Most operative events should require either assignment history, stress, loyalty condition, or relevant recent activity.
```

A hire candidate should not generate personal drama before joining the crew.

---

## 10. Affinity stacking

Yes. All matching affinities stack.

Example:

```text
Mara uses Gather Intel at The Glass Saint.

Matching:
- Gather Intel affinity
- nightlife target affinity
- maybe memory/social tag affinity if defined

All apply.
```

Important implementation rule:

> Preview and resolution must use the exact same modifier pipeline.

No “preview says one thing, resolution does another” unless randomness is explicitly involved.

---

## 11. Recruit targeting

`Recruit Operative` should target a specific hire candidate.

Remove district/venue targeting from Recruit for v0.3.

Recruit becomes:

```text
Action: Recruit Operative
Required target: hire candidate
Cost: 1 command point, 1600 resources, -4 loyalty, +3 dominion
Effect: selected candidate joins roster, candidate removed from hire pool
```

If the roster is full, the action is disabled.

Do not add firing/dismissing operatives yet unless trivial.

---

## 12. Existing saves

Version the save schema and invalidate v0.2 saves.

No migration.

Behavior:

```text
If save schema version < v0.3 schema:
  show message
  clear incompatible run
  start new v0.3 run
```

Suggested copy:

```text
This save was created with an older prototype version and is not compatible with v0.3.0 — The Roster. Start a new run to use the updated roster system.
```

This is totally acceptable at this stage. Migration is not worth the complexity yet.

---

# Final locked rules block for Codex

```text
v0.3.0 includes ten curated operatives, including Mother Neon as rare.

Rare operatives may appear in starting rosters, but at most one rare may start.

Additional rares may appear in the hire pool.

Each operative has authored starting Loyalty and Stress.

Roster cap remains five. Starting roster is three. Hire pool is four.

Signature traits and mechanical liabilities are visible immediately.

Narrative flags, exact triggers, and hidden counters remain hidden.

Breaking operatives remain assignable.

At 80+ Stress, operatives are Breaking: +10 risk and eligible for severe events.

Remove automatic compromised/unavailable behavior at 80 Stress.

Replace continuous stress risk with tier modifiers: 0 / +2 / +5 / +10.

Operative-specific events use the normal weekly event slot.

Each signature operative event can occur at most once per run.

Only current roster members can trigger operative events.

All matching affinities stack.

Preview and resolution use the same modifier pipeline.

Recruit Operative now targets a specific hire candidate only.

v0.2 saves are invalidated through save schema versioning.
```

# Why these decisions are correct

They make v0.3 about **choice under identity pressure**.

The player gets:

```text
Which crew did I roll?
Who is available to hire?
Who fits this action?
Who is too stressed?
Who has a useful but dangerous liability?
Do I spend scarce resources on a hire now?
Do I overuse my best operative and risk their event?
```

That is exactly the right depth for the next iteration.

v0.3 does not need more systems. It needs the player to start caring who is in the roster.
