## Core thesis

> **Before Haunted Apex gets more systems or a full immersion redesign, the game needs to reveal how it is actually being played and create clean places for the world to show its face.**

v0.9.0 made the game teachable through Handler guidance.

v0.9.1 should make the game **diagnosable** and **media-ready**.

This patch should not rebalance the whole game yet. It should answer:

```text
Why is Handler winning?
Which commands are actually used?
Which systems are being ignored?
Where does Dominion come from?
Where does Heat relief come from?
What is the dominant loop?
Where can images be added without blowing up the UI?
```

And visually:

```text
Can the game begin showing campaign/operative art without requiring a full app redesign?
```

# 5. Strategic Telemetry Philosophy

The game should not be tuned one command at a time blindly.

Bad balance process:

```text
Bribe an Official is underused.
Buff Bribe.

Invest in Front is underused.
Buff Fronts.

Broker Accord is underused.
Buff Accords.
```

That creates whac-a-mole.

Better balance process:

```text
What role is this command supposed to play?
In what state should it be optimal?
Is that state occurring?
When that state occurs, does Handler choose it?
If not, is the command too weak, too expensive, too opaque, or too slow?
```

v0.9.1 should gather the data needed to answer those questions.

# 6. Command Role Frame

The reports should be organized around intended command roles.

## Expand Influence

```text
Role:
Primary direct Dominion push.

Healthy use:
Common, but not sufficient alone.

Concern:
If this plus Lay Low forms the dominant winning loop, the objective is too direct.
```

## Lay Low

```text
Role:
Emergency stabilization / exposure relief.

Healthy use:
Used in crisis or to manage front/heat danger.

Concern:
If used rhythmically every other week, it is too efficient as a default Heat reset.
```

## Bribe an Official

```text
Role:
Expensive tactical Heat/security relief.

Healthy use:
Chosen when Heat/security pressure is high and Resources are available.

Concern:
If ignored even during high-Heat/resource-rich states, it is underpowered or unclear.
```

## Invest in Front

```text
Role:
Early/mid infrastructure and recurring value.

Healthy use:
Chosen early when payback horizon exists or when campaign supports fronts.

Concern:
If never chosen by Handler because the game is too short, fronts are structurally optional.
```

## Broker Accord

```text
Role:
Short-term institutional bargain.

Healthy use:
Chosen when a faction accord solves a campaign pressure or provides timed recovery.

Concern:
If ignored, accords are too slow, too costly, or less efficient than direct commands.
```

## Manage Contact

```text
Role:
Targeted social services and relationship leverage.

Healthy use:
Useful when contact services solve current pressures.

Concern:
If Dominion-raising contacts dominate all runs, contact services are too efficient.
```

## Work the Ledger

```text
Role:
Convert accumulated secrets/debts/favors into tactical swings.

Healthy use:
Used when prior setup creates a current pressure solution.

Concern:
If ignored, Ledger entries are too weak. If mandatory, Ledger is too strong.
```

# 8. Dominant Loop Detection

Add simple flags.

## Dominant pair

```ts
type CommandPairFrequency = {
  actionIds: [ActionId, ActionId];
  count: number;
  percentageOfWeeks: number;
};
```

## Loop warnings

Report warnings like:

```text
WARNING:
Expand Influence + Lay Low appears in 41% of Handler weeks.

WARNING:
Top two command types account for 67% of Handler commands.

WARNING:
Handler wins 82% of runs without establishing a new front.

WARNING:
Handler wins 91% of runs without brokering an accord.
```

These warnings do not fail the build in v0.9.1.

They are diagnostic.

# 10. Source Attribution

v0.9.1 should attribute pressure changes by source category.

## Source categories

```ts
type PressureChangeSourceKind =
  | 'action'
  | 'event'
  | 'operative'
  | 'contact'
  | 'ledger'
  | 'front'
  | 'faction'
  | 'accord'
  | 'campaign'
  | 'drift'
  | 'rival';
```

## Attribution entry

```ts
type PressureAttributionEntry = {
  week: number;
  sourceKind: PressureChangeSourceKind;
  sourceId: string;
  pressure: keyof Pressures;
  delta: number;
};
```

The implementation does not need perfect forensic granularity. It needs enough to answer:

```text
Where did Dominion come from?
Where did Heat relief come from?
Where did Resources come from?
What systems actually contributed?
```

If full attribution is difficult, start by instrumenting major resolution functions:

```text
resolve order
resolve event
apply front yield
apply accord effect
resolve contact option
resolve ledger use
apply drift
```

# 11. Media System Philosophy

v0.9.1 should not attempt to make the whole game beautiful at once.

It should introduce a reusable media layer.

The correct principle:

> **Compact cards stay playable. Detail views carry immersion.**

Do not turn the command board into an art gallery.

Use images like this:

```text
thumbnail = identity cue
hero image = immersion on detail
campaign splash = run mood
fallback = current initials/placeholder behavior
```

# 12. Media Registry

Add a static media registry.

```ts
type MediaAsset = {
  id: MediaAssetId;
  kind: 'thumb' | 'hero' | 'splash';
  src: string;
  alt: string;
  width?: number;
  height?: number;
  focalPoint?: {
    x: number;
    y: number;
  };
  tags?: string[];
};
```

Example:

```ts
export const MEDIA_ASSETS: Record<MediaAssetId, MediaAsset> = {
  op_mara_voss_thumb: {
    id: 'op_mara_voss_thumb',
    kind: 'thumb',
    src: 'assets/media/images/thumb/op-mara-voss.webp',
    alt: 'Mara Voss portrait',

  campaign_ghostline_signal_splash: {

    kind: 'splash',
    src: 'assets/media/images/splash/campaign-ghostline-signal.webp',
```

# 14. First Media Slots

## Campaign briefing splash

Add splash image support to the opening campaign briefing.

Behavior:

```text
If campaign has splashImageId:
  show splash image in briefing.

If missing:
  show existing styled briefing panel.
```

This is likely the highest visual impact per image.

## Operative card thumbnail

Replace or augment the initials box.

Behavior:

```text
If operative has thumbImageId:
  show thumbnail.

If missing:
  show initials fallback.
```

Do not enlarge the card dramatically.

## Operative detail hero

Current operative cards already open a modal/detail.

Add hero image support there.

Behavior:

```text
If heroImageId:
  show framed hero image at top of detail view.

If missing:
  show no hero or stylized placeholder.
```

## Rival detail hero, optional

If rival detail already exists or is easy:

```text
Add hero image support for rivals.
```

If not, defer to v0.9.2 or v0.10.

Keep it simple.

## Thumbnails

```text
Can load on card render.
Small enough to be fine.
```

## Hero images

```text
Load only when detail modal opens.
```

## Campaign splash

```text
Load when briefing opens.
```

No complex preloading required in v0.9.1.

---

# 18. UI Design Rules for Images

## Do

```text
Frame images inside existing panels.
Use gradient overlays if text sits on image.
Keep thumbnails small and consistent.
Use object-fit: cover.
Use fallback initials.
Use alt text.
```

## Do not

```text
Stretch images full-screen on desktop.
Redesign every panel.
Make cards huge.
Let image loading break layout.
Require every entity to have art.
```

v0.9.1 should prove visual integration, not finish it.
