# Haunted Apex v0.9.1 — Clarifications

## 1. Telemetry scope

For v0.9.1, **Strategic Fingerprint should be harness/debug-first**, not a major player-facing feature.

Player-facing run summary can get a compact hook only if low-risk.

Locked rule:

```text
Strategic Fingerprint is primarily harness/debug output in v0.9.1.
```

Allowed player-facing addition:

```text
Completed Run Summary:
- Top command used
- Main Dominion source
- Main Heat relief source
- Systems used: Fronts / Accords / Ledger / Contacts
```

But do not turn the post-run summary into a full analytics dashboard yet. That can come later once the report format stabilizes.

## 2. Analytics privacy

Correct: **fully local only**.

No remote telemetry.

No GitHub Pages event collection.

No user tracking.

No analytics beacon.

No external reporting.

Locked rule:

```text
All v0.9.1 analytics are local simulation/reporting only.
```

This is development telemetry and run analysis, not product analytics.

Player privacy posture remains simple:

```text
The game runs locally in the browser.
No gameplay analytics are transmitted.
```

## 3. Report format

Keep the current multi-section readable harness output.

Add sections like:

```text
strategic_fingerprint
command_usage
command_pairs
source_breakdown
system_engagement
loop_warnings
```

Do not require a JSON export in v0.9.1.

Locked rule:

```text
v0.9.1 extends the existing readable/CSV-style harness output.
```

JSON export is a good future enhancement, especially if reports get large or you want charts later, but it is not necessary for this patch.

Suggested section shape:

```text
[strategic_fingerprint]
bot,campaign,runs,wins,total_commands,top_command,top_pair,dominant_loop_warning

[command_usage]
bot,campaign,action_id,count,percentage

[command_pairs]
bot,campaign,action_a,action_b,count,percentage_of_weeks

[source_breakdown]
bot,campaign,pressure,source_kind,source_id,total_delta

[system_engagement]
bot,campaign,runs_with_fronts,runs_with_accords,runs_with_ledger,runs_with_contacts,runs_with_bribe,runs_with_lay_low

[loop_warnings]
bot,campaign,warning_type,value,threshold,message
```

## 4. Source attribution precision

Use **major-source attribution** first.

Do not make v0.9.1 surgically track every micro-delta.

Locked rule:

```text
v0.9.1 source attribution should be good enough for strategic diagnosis, not perfect forensic accounting.
```

Acceptable:

```text
Resolved command delta mostly attributed to action.
Front yield attributed to front.
Accord effects attributed to accord.
Ledger use attributed to ledger.
Contact option attributed to contact.
Event choice attributed to event.
Drift attributed to drift.
```

Do not split every action into:

```text
base action
operative modifier
target modifier
campaign modifier
trait modifier
affinity modifier
```

unless the current engine already makes that trivial.

Operative contribution can be tracked separately through engagement metrics:

```text
operative assignment count
operative used in winning runs
operative stress outcomes
operative-associated command outcomes
```

The main questions for v0.9.1 are:

```text
Where did Dominion come from?
Where did Heat relief come from?
Which command/system did Handler rely on?
```

Major-source attribution answers those.

---

## 5. Player runs

Design the telemetry model so player runs can use it later, but implement reporting against **harness runs first**.

Locked rule:

```text
v0.9.1 reporting targets bot/harness runs.
```

The model can still be generic:

```ts
type RunTelemetry = {
  runId: string;
  actorType: 'bot' | 'player';
  botId?: BotId;
  campaignTensionId: CampaignTensionId;
  entries: TelemetryEntry[];
};
```

But the UI does not need to expose full telemetry for live human play yet.

This keeps scope tight while avoiding throwaway design.

## 6. Media assets path

Use the current Angular/public setup.

Since your current config copies from `public/`, use:

```text
public/assets/media/images/...
```

not:

```text
src/assets/...
```

Locked rule:

```text
v0.9.1 media assets live under public/assets/media/images.
```

Recommended structure:

```text
public/assets/media/images/
  splash/
    campaign-corp-crackdown.webp
    campaign-nightlife-war.webp
    campaign-ghostline-signal.webp
    campaign-industrial-cut.webp
    campaign-dirty-capital.webp

  thumb/
    op-mara-voss.webp
    op-juno-hex.webp
    op-saint-calder.webp

  hero/
    op-mara-voss.webp
    op-juno-hex.webp
    op-saint-calder.webp
```

The media registry should reference runtime paths like:

```ts
src: 'assets/media/images/thumb/op-mara-voss.webp';
```

not filesystem paths.

## 7. First media slice

Code should support all planned media slots, but the release does **not** need to ship with the full 5 campaign + 10 operative set.

Locked rule:

```text
v0.9.1 code supports the full first-wave media map, but committed asset count can be lower.
```

Minimum useful asset set for the patch:

```text
1 campaign splash
1–3 operative thumbnails
1–3 operative hero images
```

Ideal first slice:

```text

10 operative thumbs

```

But don’t block the patch on you generating all of them.

The important implementation requirement is:

```text
Missing media falls back cleanly.
```

Locked rule:

## 9. Handler validation

Keep `npm run validate:handler` as a **local/manual release gate** for now.

Do not add heavy non-browser validation to automated CI in v0.9.1.

Locked rule:

```text
Handler validation remains a documented local/manual release gate.
```

Add to release checklist:

```text
Before tagging v0.9.1:
- npm run test
- npm run build
- npm run validate:handler
- review strategic fingerprint report

```

CI can keep doing lighter checks:

```text

```

Heavy 500-run handler validation can stay local until the runtime and CI budget are better understood.
