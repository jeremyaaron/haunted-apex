# Haunted Apex v0.4.0 — Black Ledger Clarifications

## 1. Should Favors be fully in v0.4, or optional?

Implement all three:

```text
Secret
Debt
Favor
```

But make **Favors the thinnest type**.

For v0.4, Favors do not need deep event chains or custom logic. They can simply be beneficial, consumable Ledger entries that act as clean-ish comeback tools.

Example:

```text
Favor: Checkpoint Captain
Use: Heat -10, Intel -2
Consumes Favor
```

This keeps the conceptual model clean. If the UI has three tabs — Secrets, Debts, Favors — all three should actually exist.

---

## 2. Should Work the Ledger consume Command like any other order?

Yes.

```text
Work the Ledger
Command Cost: 1
Queued during Command Phase
Resolved during Advance Week
```

It should behave like a normal action, not an instant button. That keeps the action economy meaningful.

The player should have to choose:

```text
Do I expand influence, recruit, run a job, or spend this week cashing in a secret?
```

That is the whole point.

---

## 3. Can Work the Ledger have Resource or Intel costs through the targeted entry?

Yes.

The `Work the Ledger` action itself has no default cost except command cost, but the **targeted Ledger entry/use option** can impose costs.

Example:

```text
Settle Debt: Owes the Liaison
Cost: Resources -900
Effect: Resolve debt, Loyalty +2
```

or:

```text
Settle Debt: Owes the Liaison
Cost: Intel -5
Effect: Resolve debt, Heat -2
```

The preview should include both command cost and entry-specific costs.

---

## 4. For debts with multiple settlement options, should the player choose the settlement method?

Yes.

Do not hide the settlement method inside the engine.

For v0.4, the cleanest implementation is probably to model Ledger uses as explicit sub-options.

Example UI:

```text
Debt: Owes the Liaison

Available Ledger Uses:
- Pay in Credits
  Cost: Resources -900
  Effect: Resolve Debt, Loyalty +2

- Offer Information
  Cost: Intel -5
  Effect: Resolve Debt, Heat -2

- Refuse for Now
  No action
```

Implementation shape:

```ts
type LedgerUseOption = {
  id: string;
  label: string;
  cost?: PressureCost;
  effects: PressureDelta;
  consumesEntry: boolean;
};
```

Then `Work the Ledger` targets both:

```ts
type ActionTarget = { type: 'ledger'; entryId: LedgerEntryId; useOptionId: LedgerUseOptionId };
```

If that is too cumbersome for the first pass, generate separate selectable target options in the UI:

```text
Owes the Liaison — Pay Resources
Owes the Liaison — Offer Intel
```

Mechanically, both are fine. The important thing is that the player chooses the method before queueing.

---

## 5. Should unresolved Debts directly cause loss conditions in v0.4?

No.

No new hard loss condition.

Debts should affect:

```text
event weighting
pressure damage
run summary
future consequence
player anxiety
```

But not:

```text
Debt count >= 3 → instant loss
```

v0.4 is too early for that. Debts should feel ominous and strategically relevant, not like a hidden alternate death meter.

---

## 6. Should consumed Ledger entries stay visible?

Yes.

Consumed entries should remain visible as history.

Recommended UI:

```text
Active
Spent / Resolved
```

Spent/resolved entries should be visually muted or collapsed by default.

Do not delete them from the run. The whole point of the Black Ledger is memory.

---

## 7. Should Gather Intel discover Secrets only when targeted?

Yes.

Strict rule:

```text
Only targeted Gather Intel can discover a Secret.
```

Untargeted Gather Intel remains the clean meter action:

```text
+ Intel
+ Heat
no Secret discovery
```

This preserves v0.2’s target-choice thesis. If the player wants named secrets, they must look somewhere specific.

---

## 8. Should Secret discovery be previewed as a probability?

Yes.

Show it qualitatively or numerically. For v0.4, I’d show the percentage because it is a direct probabilistic reward and your game already exposes enough math to be strategic.

Example:

```text
Secret Chance: 28%
```

Or if you want softer language:

```text
Secret Chance: Moderate
```

My recommendation:

```text
Show exact percentage in preview for now.
```

This helps tuning and makes targeted Intel feel concrete. Resolution remains seeded/random.

---

## 9. Should event choices preview exact Ledger entries?

Yes, absolutely.

Use:

```text
Creates Debt: Owes the Liaison
Creates Secret: Stored Blackmail
Creates Favor: Checkpoint Captain
Consumes Secret: Patrol Schedule
Resolves Debt: Unfunded Promise
```

Not vague copy like:

```text
Creates a Debt
```

This avoids gotchas and makes the Ledger feel like a real system.

---

## 10. Should Ledger-specific events select a specific active entry and show it in the event copy?

Yes.

Prefer specific events over generic ones.

Better:

```text
Debt Comes Due: Owes the Liaison
```

than:

```text
Debt Comes Due
```

Example:

```text
Debt Comes Due: Owes the Liaison

The liaison has stopped using intermediaries. The message arrives in their own voice this time.
```

The event should carry the selected `ledgerEntryId` so choices can resolve or modify that specific entry.

---

## 11. Should duplicate Ledger entries be allowed?

Yes, allow duplicates as separate instances.

Example:

```text
Debt: Contaminated Money, created Week 2
Debt: Contaminated Money, created Week 5
```

They are separate obligations.

But prevent noise through weighting:

```text
same definition already active → reduce generation weight
same debt active twice → strongly reduce further duplicates
```

Do not hard-block duplicates unless a specific entry is conceptually unique.

Good rule:

```text
Allow duplicates by default.
Mark specific definitions as unique if needed.
```

Example:

```ts
type LedgerEntryDefinition = {
  id: string;
  unique?: boolean;
  repeatWeightMultiplier?: number;
};
```

## 13. Should run-end summary appear only after game over, or during active run?

Only after victory/loss for v0.4.

Add it to the game-over panel.

Include:

```text
Copy Run Report
```

Do not build an always-available run history dashboard yet.

That can come later.

---

## 14. Should v0.3 saves be invalidated?

Yes.

Version the save schema and invalidate older saves.

Suggested behavior:

```text
Detected save from v0.3.x.
v0.4.0 changes the game state schema and requires a fresh run.
Start New Run.
```

No migration yet.

# Locked v0.4 Rules Block for Codex

```text
Implement all three Ledger kinds: Secret, Debt, Favor.

Favors are thin consumable benefit entries in v0.4.

Work the Ledger costs 1 command point and resolves during Advance Week.

Work the Ledger targets a specific Ledger entry and use option.

Ledger use options may have Resource, Intel, or pressure costs.

If a Ledger use option is unaffordable, queueing is disabled.

Debts with multiple settlement methods require the player to choose a method.

Unresolved Debts do not create a hard loss condition in v0.4.

Consumed Ledger entries remain visible in a muted/collapsed Spent or Resolved section.

Only targeted Gather Intel can discover Secrets.

Targeted Gather Intel preview shows Secret discovery chance.

Event choices preview exact Ledger entries created, consumed, or resolved.

Ledger-specific events select a specific active Ledger entry where possible.

Duplicate Ledger entries are allowed as separate instances.

Definitions may opt into unique behavior later.

Initial content target is 12 entries: 6 Secrets, 4 Debts, 2 Favors.

Run-end summary appears only after victory/loss.

Copy Run Report appears on the game-over panel.

v0.3 saves are invalidated through save schema versioning.
```

# The most important implementation principle

For v0.4, the Ledger must be **visible before it is punishing**.

Players should always understand:

```text
what they gained
what they owe
what can be spent
what might come due
```

The system can be dangerous. It should not be sneaky.

That keeps the Black Ledger in the Haunted Apex tone:

> Power with receipts. Consequences with style.
