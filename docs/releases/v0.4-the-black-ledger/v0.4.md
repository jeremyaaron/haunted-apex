# Haunted Apex v0.4.0 — The Black Ledger

## Foundation Vision Document

## Version

```text
v0.4.0 — The Black Ledger
```

## Previous milestone

```text
v0.3.0 — The Roster
```

## Core thesis

> **Intel should stop being only a number and become playable memory: secrets, debts, favors, blackmail, promises, and receipts.**

v0.1 proved the pressure loop.

v0.2 made territory and rivals matter.

v0.3 made operatives matter.

v0.4 should make **information and obligation** matter.

The player should no longer think only:

```text
I have 22 Intel.
```

They should think:

```text
I know what Nyx is hiding.
I owe the liaison a favor.
I have a patrol schedule I can burn to lower Heat.
I promised the crew money I do not have.
I can cash in this secret now, but it will make the city uglier.
```

The goal is not a full relationship system yet. It is to introduce the first real **memory layer**.

---

# 1. Design Goal

v0.4 introduces the **Black Ledger**: a visible collection of leverage and liabilities accumulated during a run.

The Ledger should make the campaign feel less like a series of disconnected weekly events and more like a history of choices coming due.

## v0.3 structure

```text
Action + Operative Identity + Operative Condition + Target
→ Outcome + Consequence
```

## v0.4 structure

```text
Action + Operative + Target + Ledger State
→ Outcome + Consequence + Future Hook
```

The key addition is **future hooks**.

Some choices should leave artifacts behind:

```text
Secret acquired
Debt accepted
Favor banked
Promise made
Evidence hidden
Witness spared
Asset burned
```

Those artifacts should matter later.

# 2. What v0.4 Should Prove

v0.4 should answer:

> **Can Haunted Apex make player choices feel remembered?**

Examples of target player stories:

```text
“I took contaminated money in Week 3, and the debt came due right before the final push.”

“I saved blackmail on a magistrate for four weeks and used it to survive the Corp Patrol Sweep.”

“I kept accepting favors from the liaison and ended the game with great Intel but a disgusting amount of debt.”

“I had enough Dominion to win, but my Black Ledger was full of unresolved obligations and the city ate me.”

“I used Nyx’s secret against her, but it spiked Ruin and triggered Juno’s event.”
```

That is the desired texture.

# 5. Content Philosophy

The Black Ledger is not an inventory system.

It is a **consequence system**.

A good Ledger entry should be:

```text
specific
useful
dangerous
memorable
connected to prior choices
```

Bad Ledger entry:

```text
Generic Intel Token
+3 Dominion
```

Good Ledger entry:

```text
Magistrate in the Glass Room
A corp magistrate was recorded entering a memory lounge under a false name.
Use: reduce Heat or gain Dominion.
Cost: increases Ruin and rival attention.
```

The Ledger should feel like the city has receipts.

# 6. Core Ledger Types

v0.4 should support three kinds of entries.

```ts
type LedgerEntryKind = 'secret' | 'debt' | 'favor';
```

## Secret

A beneficial piece of leverage.

Examples:

```text
blackmail
patrol schedule
hidden route
rival weakness
corp access key
buyer list
witness testimony
memory recording
```

Secrets are usually gained through:

```text
Gather Intel
event choices
operative events
targeting specific venues/rivals
```

Secrets can be spent through:

```text
event choices
Work the Ledger action
future targeted actions
```

## Debt

A liability the player accepted.

Examples:

```text
owed liaison favor
unfunded promise to crew
contaminated money
rival courtesy accepted
Saint’s concealed payment trail
ghost-market obligation
```

Debts usually help now and hurt later.

They should not be pure punishment. They are **dirty financing**.

## Favor

A positive obligation owed to the player.

Examples:

```text
checkpoint captain owes you
liaison promised a room
operative contact can hide evidence
rival subordinate owes silence
Ghostline broker owes a route
```

Favors are comeback tools.

They should be useful, but not infinitely stackable.

# 7. Ledger Entry Model

## Definition

Static definitions live in content.

```ts
type LedgerEntryDefinition = {
  id: LedgerEntryDefinitionId;
  kind: LedgerEntryKind;

  name: string;
  description: string;

  rarity: 'common' | 'uncommon' | 'rare';

  tags: string[];

  defaultUse?: LedgerUseDefinition;
  triggerProfile?: LedgerTriggerProfile;

  flavor?: {
    shortLabel?: string;
    visualTags?: string[];
  };
};
```

## Instance

Runtime entries live in `GameState`.

```ts
type LedgerEntry = {
  id: LedgerEntryId;
  definitionId: LedgerEntryDefinitionId;

  kind: LedgerEntryKind;

  createdWeek: number;
  source: LedgerEntrySource;

  potency: number; // 1–3 for v0.4
  revealed: boolean;

  consumed: boolean;

  relatedTarget?: ActionTarget;
  relatedOperativeId?: OperativeId;
  relatedRivalId?: RivalId;

  flags?: Record<string, boolean | number | string>;
};
```

## Source

```ts
type LedgerEntrySource =
  | {
      type: 'event';
      eventId: EventId;
      choiceId: EventChoiceId;
    }
  | {
      type: 'action';
      actionId: ActionId;
      target?: ActionTarget;
    }
  | {
      type: 'operative_event';
      operativeId: OperativeId;
      eventId: EventId;
    };
```

```ts
type GameState = {
```

Expected v0.4 volume:

```text

```

Do not overpopulate the Ledger.

Each entry should matter.

# 9. New Action: Work the Ledger

Add one new action.

```text
Work the Ledger
```

This action targets a Ledger entry.

It is the general “use or resolve this thing” action.

```ts
type ActionTarget =
  | { type: 'district'; id: DistrictId }
  | { type: 'venue'; id: VenueId }
  | { type: 'rival'; id: RivalId }
  | { type: 'recruit'; id: OperativeId }
  | { type: 'ledger'; id: LedgerEntryId };
```

## Action definition

```ts
{
  id: 'work_the_ledger',
  label: 'Work the Ledger',
  commandCost: 1,
  resourceCost: 0,
  effects: {},
  baseRisk: 12,
  requiresTarget: true,
  allowedTargetTypes: ['ledger']
}
```

The effects are determined by the targeted Ledger entry.

## Using a Secret

Example:

```text
Use Secret: Magistrate in the Glass Room

Effects:
- Heat -10
- Dominion +4
- Ruin +2
- Consumes secret
```

## Settling a Debt

Example:

```text
Settle Debt: Owes the Liaison

Cost:
- Resources 900 or Intel 5

Effects:
- Loyalty +2
- Reduces future liaison/debt event weight
- Consumes debt
```

## Calling in a Favor

Example:

```text
Call Favor: Checkpoint Captain

Effects:
- Heat -12
- Intel -2
- Favor consumed
```

## UI copy

```text
Work the Ledger
Turn secrets into leverage, settle dangerous obligations, or call in favors before they expire into consequences.
```

# 10. Ledger Discovery

## Gather Intel Update

`Gather Intel` should have a chance to discover a Secret when targeted.

Discovery should depend on:

```text
target tags
operative affinities
Intel pressure
district secrecy
venue intelMod
rival pressure
```

Simple v0.4 rule:

```ts
function shouldDiscoverSecret(state, order, resolution): boolean {
  if (order.actionId !== 'gather_intel') return false;
  if (!order.target) return false;

  const base = 18;
  const intelBonus = Math.floor(state.pressures.intel / 10);
  const targetBonus = getTargetIntelDiscoveryBonus(order.target, state);
  const stressPenalty = getAssignedOperativeStressPenalty(order, state);

  const chance = clamp(base + intelBonus + targetBonus - stressPenalty, 5, 45);

  return rng.rollPercent(chance);
}
```

If a secret is discovered:

```text
Add Ledger entry
Log discovery
Possibly reduce raw Intel gain slightly, if balance requires
```

Initial recommendation:

```text
Do not reduce raw Intel gain yet.
Tune later if Intel becomes too strong.
```

# 11. Existing Event Integration

v0.4 should update existing events so important choices leave Ledger entries.

## Liaison Offers a Favor

Current likely behavior:

```text
Accept favor → Intel/Dominion/Ruin + owes_liaison flag
```

v0.4:

```text
Accept favor → create Debt: Owes the Liaison
Pay instead → no debt
Decline → no debt, loyalty/stability benefit
```

## Blackmail Lead Emerges

v0.4:

```text
Exploit immediately → immediate effects, no Ledger secret
Save it for later → create Secret: Stored Blackmail
Trade it quietly → resources now, lose Intel/no secret
```

## Unexpected Windfall

v0.4:

```text
Take it → resources now + Debt: Contaminated Money
Trace it first → smaller resources + Secret: Dead Channel Trace
Refuse → loyalty benefit/no Ledger
```

## Operative Wants More

v0.4:

```text
Pay them → no debt
Promise future rewards → Debt: Unfunded Promise
Remind them who built this → no debt, but Ruin/Loyalty damage
```

## Corp Patrol Sweep

v0.4 optional integration:

```text
Feed them a rival name → create Secret or Debt depending implementation:
- Secret: False Names in the Sweep
or
- Debt: Fabricated Evidence
```

Recommendation:

```text
Do not over-integrate every event in v0.4.
Update 4–5 existing events first.
```


```text
At least one active Secret.
More likely if Dominion < 45 or Heat >= 65.
```

```text
Leverage Window

A door opens for exactly the wrong reason. The secret will work now, or it will rot.
```

Choices:

```ts
[
```

# 17. Agent Updates

The five agents should learn basic Ledger behavior.

Existing agents:

```text
random
greedy
aggressive
cautious
operator
```

## OperatorBot

Should:

```text
Use Secrets when they solve current pressure problems.
Settle Debts if Debt risk is high and resources/intel allow.
Call Favors when Heat or Loyalty is in danger.
Avoid creating too many Debts unless needed.
Prefer discovering Secrets if behind but not dying.
```

Example logic:

```text
If Heat >= 75 and active Secret/Favor can reduce Heat, Work the Ledger.
If active Debt age >= 2 and resources > 1800, settle it.
If Dominion behind schedule and Secret can add Dominion, use it.
If no active Ledger entries and Intel strategy is viable, Gather Intel at high-secrecy target.
```

## CautiousBot

Should:

```text
Avoid creating Debts.
Settle Debts early.
Use Secrets primarily to lower Heat.
Likely still fails Dominion target.
```

## AggressiveBot

Should:

```text
Exploit Secrets for Dominion.
Ignore Debts until severe.
Accept dirty Ledger choices.
```

## GreedyBot

Should:

```text
Use Secrets/Favors for Resources.
Accept contaminated money.
Delay debt settlement.
```

## RandomBot

Random valid Ledger behavior.


No Ledger entry should be universally correct to use immediately.

No Debt should be so harmless that players always accept it.

# 20. Regret Patterns in v0.4

## Good regret

```text
I accepted the liaison debt because I needed Intel, and it came due when I was broke.

I sold a secret for money, then had no leverage during the patrol sweep.

I hoarded blackmail too long and lost before using it.

I settled a debt early and later wished I had used that money for recruitment.

I used a secret to win, but Ruin spiked and Juno broke.
```

## Bad regret

```text
A hidden Debt killed me with no warning.

The UI did not tell me a choice created a liability.

A Secret disappeared without explanation.

Work the Ledger preview said one thing and resolution did another.

The event pool became all debt events all the time.
```

The Ledger should create consequences, not gotchas.

# 21. Technical Guidance

## Avoid hardcoded event flags

v0.4 should replace some ad hoc flags with Ledger entries.

Bad:

```ts
flags.owes_liaison = true;
```

Better:

```ts
addLedgerEntry(state, {
  definitionId: 'debt_owes_liaison',
  source: {
    type: 'event',
    eventId,
    choiceId,
  },
});
```

## But do not migrate every flag yet

Some flags can remain.

The point is to formalize meaningful, player-facing consequences.

## Suggested modules

```text
content/ledgerEntries.ts
engine/ledger/addLedgerEntry.ts
engine/ledger/useLedgerEntry.ts
engine/ledger/selectDiscoverableSecret.ts
engine/ledger/ledgerSelectors.ts
engine/reports/runSummary.ts
```

## Shared preview/resolution pipeline

As with operative affinities, preview and resolution must use the same use-effect pipeline.

```ts
previewLedgerUse(state, ledgerEntryId);
resolveLedgerUse(state, ledgerEntryId);
```

The preview should be deterministic except for explicitly labeled risk.

# 26. Summary

v0.4 should make Haunted Apex feel like the city remembers what the player did.

v0.1 asked:

```text
Can the pressure loop work?
```

v0.2 asked:

```text
Where do I dare do this?
```

v0.3 asked:

```text
Who do I trust to do it?
```

v0.4 asks:

```text
What have I learned, what do I owe, and what happens when I use it?
```

That is the right next layer.

The game already has actions, places, rivals, and people.

Now it needs receipts.
