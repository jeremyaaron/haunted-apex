Here’s the v0 spec I’d give Codex.

# Haunted Apex v0 — Playable UI & Turn Model

## 1. What are the actual screens for v0?

For v0, use **one main screen** with internal panels/tabs, not multiple routes.

Screen name:

```text
The Black Ledger
```

It should contain:

```text
Top bar:
- Week 1 / 8
- Command points remaining
- Win/loss status
- Advance Week button

Left panel:
- Pressure meters
  - Dominion
  - Heat
  - Loyalty
  - Resources
  - Intel
  - Ruin

Center panel:
- Available actions
- Queued actions for this week

Right panel:
- Operative roster
- Selected operative / assignment state

Bottom panel:
- Event feed / resolution log
```

Optional internal tabs inside the same screen:

```text
Command
Operatives
Log
Debug
```

But don’t create a full app nav yet.

For v0, the game should feel like:

> “I can see the whole prototype state at once.”

That makes balancing easier.

# 2. Is the first playable UI one dashboard, or multiple tabs?

**One dashboard.**

Use internal tabs only if needed.

Recommended v0 layout:

```text
[Header: Week / CP / Advance Week]

[Pressure Meters] [Action Cards] [Operatives]

[Queued Orders]

[Event Log / Weekly Fallout]
```

Do **not** build separate screens for:

```text
City Map
Relationships
Factions
Inventory
Settings
Districts
```

Those are later.

For v0, the dashboard is the game.

# 3. How deterministic should runs be?

Use **seeded determinism**.

Every new game has a seed:

```ts
seed: string;
rngCursor: number;
```

Same seed + same player choices = same outcome.

This is important for:

- debugging
- tuning
- sharing weird runs
- Codex/test reproducibility
- future balance bots

New game config:

```ts
type NewGameConfig = {
  seed?: string;
  difficulty: "standard";
};
```

If no seed is provided, generate one.

Show seed in debug panel:

```text
Seed: VIOLET-ASH-1047
```

For the player-facing v0, randomness can feel mysterious. For the dev, it must be reproducible.

---

# 4. Are events chosen before or after actions?

For v0:

> **Events are chosen after actions resolve.**

Turn order:

```text
1. Player queues up to 2 actions.
2. Player clicks Advance Week.
3. Queued actions resolve.
4. Action risks/complications resolve.
5. Weekly drift applies.
6. One weekly event is selected based on updated state.
7. Event choice is presented.
8. Player chooses event response.
9. Event effects apply.
10. Win/loss is checked.
11. Week increments.
```

This is better than choosing events before actions because events can respond to what the player just did.

Example:

```text
Player runs two Small Jobs.
Heat rises.
Now “Job Goes Loud” or “Corp Patrol Sweep” becomes more likely.
```

That makes the city feel reactive.

---

# 5. Can players preview action outcomes?

Yes. For v0, players should see **base predicted effects** and **risk category**, but not exact hidden rolls.

Action card should show:

```text
Cost:
- 1 Command
- 400 Resources

Expected:
+10 Intel
+2 Heat

Risk:
Low

Requires:
Operative optional / recommended: Subtlety
```

If an operative is selected, preview adjusted effects:

```text
Assigned: Mara Voss

Adjusted Expected:
+10 Intel
+1 Heat

Risk:
Very Low
```

Use qualitative risk labels:

```ts
type RiskLabel = "Very Low" | "Low" | "Moderate" | "High" | "Severe";
```

Mapping:

```ts
function riskLabel(chance: number): RiskLabel {
  if (chance <= 6) return "Very Low";
  if (chance <= 14) return "Low";
  if (chance <= 24) return "Moderate";
  if (chance <= 34) return "High";
  return "Severe";
}
```

Do not show:

```text
17.384% chance of counter-surveillance
```

That feels too spreadsheety.

Show:

```text
Exposure Risk: Moderate
Complication Risk: Low
```

Later you can add settings for exact numbers.

# 6. Do risks trigger immediately per action, or during weekly resolution?

During weekly resolution.

The player queues orders, then the week plays out.

So risk does **not** trigger immediately when they click an action card. It triggers after **Advance Week**.

Why:

- creates anticipation
- makes “Advance Week” meaningful
- keeps the game loop clean
- allows combined consequences
- avoids click-by-click whiplash

V0 model:

```text
Queue action → preview effects → assign operative → Advance Week → resolve all
```

Resolution order for queued actions:

```ts
for (const order of queuedOrders) {
  resolveOrderBaseEffects(order);
  resolveOrderRisk(order);
  applyOperativeStress(order);
}
```

Then weekly drift.

Then weekly event.

---

# 7. What does “assigned” mean if there are 2 command points and 3 operatives?

For v0:

> Each queued action may have zero or one assigned operative.
> An operative can be assigned to only one action per week.

So with:

```text
2 command points
3 operatives
```

The player can do:

```text
Action 1: Gather Intel with Mara
Action 2: Bribe Official with Saint
Juno: idle
```

Or:

```text
Action 1: Lay Low with no operative
Action 2: Gather Intel with Juno
Mara/Saint: idle
```

Assigned means:

```text
The operative modifies that action’s effects/risk and gains stress.
```

Idle operatives:

```text
gain no stress
recover 2 stress passively if stress > 0
```

Assigned operatives:

```text
gain action stress, usually +6
dangerous actions +10
```

Lay Low special case:

```text
Lay Low does not require an operative.
If one is assigned, that operative recovers 8 stress.
```

This creates an early meaningful choice:

> Do I use all people every week for maximum optimization, or let someone recover?

---

# 8. Does recruitment increase command capacity or only specialist options?

For v0:

> Recruitment increases specialist options, not command capacity.

Keep command points fixed at 2.

Why:

- easier to balance
- avoids snowballing
- keeps action economy tight
- makes recruitment a qualitative choice, not always correct

Recruitment benefits:

```text
More skill coverage
More backup if someone is injured/compromised
More ways to modify actions
More resilience against stress
```

Recruitment costs:

```text
-1600 Resources
-4 Loyalty
+3 Dominion
```

Roster cap:

```text
maxOperatives = 5
```

Later, command capacity can be unlocked by infrastructure:

```text
Command Center upgrade
Trusted lieutenant
Second safehouse
District control milestone
```

But not v0.

This is important: if recruitment gives command points, it becomes the obvious dominant early strategy.

# Final v0 turn model

Codex can implement this exact flow:

```ts
type TurnPhase =
  | "COMMAND"
  | "RESOLVING_ACTIONS"
  | "EVENT_CHOICE"
  | "WEEK_COMPLETE"
  | "GAME_OVER";
```

## Command phase

Player may queue actions while:

```ts
commandPointsRemaining > 0;
```

Each queued order:

```ts
type QueuedOrder = {
  id: string;
  actionId: ActionId;
  assignedOperativeId?: string;
};
```

Rules:

```text
- Max 2 command points per week.
- Each action costs 1 command point.
- Each operative can be assigned once.
- Some actions may allow no operative.
- Queued orders can be removed before Advance Week.
```

## Advance Week

```ts
function advanceWeek(state: GameState): ResolutionResult {
  let next = clone(state);

  for (const order of next.queuedOrders) {
    next = resolveQueuedOrder(next, order);
  }

  next = applyWeeklyDrift(next);
  const event = selectWeeklyEvent(next);
  next.pendingEvent = event;
  next.queuedOrders = [];

  return {
    state: next,
    phase: "EVENT_CHOICE",
    logEntries,
  };
}
```

## Event choice

Player chooses one response.

```ts
function resolveEventChoice(
  state: GameState,
  eventId: string,
  choiceId: string,
): GameState {
  let next = applyEventChoiceEffects(state, eventId, choiceId);
  next = checkWinLoss(next);

  if (!next.gameOver) {
    next.week += 1;
    next.phase = "COMMAND";
  }

  return next;
}
```


```ts
queuedOrders.length > 0 && phase === "COMMAND";
```

# V0 design answers in one block

```text
Screens:
One screen: The Black Ledger, with panels/tabs.

Tabs:
Command, Operatives, Log, Debug if needed.

Determinism:
Seeded deterministic. Same seed + same choices = same run.

Event timing:
Actions resolve first. Weekly event selected after actions and drift.

Preview:
Players see base/adjusted effects and qualitative risk.

Risk timing:
Risks resolve during Advance Week, not immediately on action click.

Assignment:
One operative per action, one action per operative per week.

Command points:
2 per week, fixed in v0.

Recruitment:
Adds specialist options and resilience, not command capacity.

Idle:
Idle operatives recover 2 stress.

Lay Low:
Can be unassigned; assigned operative recovers 8 stress.

Victory:
Dominion >= 60 by end of Week 8.

Loss:
Heat >= 100, Loyalty <= 0, or Resources < 0.
```

The main thing I’d be strict about: **do not let v0 sprawl into multiple screens or expandable systems yet.** Make one dashboard where the whole turn loop is visible. Once that works, the city can grow.
